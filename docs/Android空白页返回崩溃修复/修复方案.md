# Android 工作区空白页返回崩溃 - 修复方案

| 文档项 | 内容 |
|---|---|
| 流程级别 | L2 - Bug 修复 |
| 版本 | v1.0 |
| 日期 | 2026-05-18 |
| 涉及端 | Android |

## 1. 问题描述

### 现象
进入某个工作区（消息列表为空 / 仍在 loading），立即点返回回到主页，弹出红屏：
```
'package:flutter/src/rendering/object.dart': Failed assertion:
line 2265 pos 12: '_owner != null': is not true.
```

### 期望行为
进入空工作区再返回，应平滑回到主页，不崩溃。

### 复现步骤
1. 启动 Android 客户端，绑定 Mac 成功
2. 主页 tab 列表点击任一工作区
3. 进入工作区瞬间（消息列表还没加载出来）立即点左上角返回箭头
4. → 红屏报错

## 2. 根因分析

logcat stack trace（关键 3 行）：
```
#0  ScrollPosition.maxScrollExtent  (scroll_position.dart:239:49)
#1  _MessageCardListState._jumpToBottom  (message_card_list.dart:115:45)
#2  _MessageCardListState._scheduleInitialJump.<anonymous closure>  (message_card_list.dart:77:9)
#3  new Future.delayed.<anonymous closure>  (dart:async/future.dart:422)
```

问题代码 `message_card_list.dart:73-79`：
```dart
void _scheduleInitialJump() {
    for (final ms in const [0, 80, 200, 400]) {
      Future.delayed(Duration(milliseconds: ms), () {
        if (!mounted) return;
        _jumpToBottom();
      });
    }
```

`_jumpToBottom`（行 113-116）：
```dart
void _jumpToBottom() {
    if (!_scrollCtrl.hasClients) return;
    _scrollCtrl.jumpTo(_scrollCtrl.position.maxScrollExtent);  // ← 这里崩
}
```

**触发链：**
1. 用户进入聊天页 → `MessageCardList.initState` → `_scheduleInitialJump` 安排 4 个 `Future.delayed`（0/80/200/400ms）+ 1 个 450ms 的 `_initialScrollDone` 标记
2. 用户**立即返回**（在 450ms 内）
3. ChatScreen.dispose → MessageCardList unmount → ListView 的 RenderObject 从 pipeline owner detach
4. 但 _MessageCardListState 的 dispose 此时**还未执行**，`mounted` 仍为 true；ScrollController 也没 dispose，`hasClients` 也为 true
5. 此时某个 timer fire → mounted check 通过 → hasClients check 通过 → 访问 `position.maxScrollExtent` 内部 assertion `_owner != null` 失败 → 红屏

**根本原因：** `Future.delayed` 创建的 timer 没法被取消，dispose 后 closure 仍会执行；而 `hasClients` + `mounted` 这两个检查不足以覆盖"RenderObject 已 detach 但 State/Controller 还活着"的中间态。

## 3. 修复方案

**改成可取消的 `Timer`**，在 `dispose` 时主动 cancel 所有 pending timer。这从根上消除"已 unmount 后 closure 仍触发"的窗口。

代码改动（仅 `AndroidClient/lib/features/chat/widgets/message_card_list.dart`）：

1. 顶部加 `import 'dart:async';`（如未导入）
2. 加字段：`final List<Timer> _initJumpTimers = [];`
3. `_scheduleInitialJump` 把 `Future.delayed` 替换为 `Timer(...)` 并存入数组
4. `dispose` 中先把 `_initJumpTimers` 全部 `cancel()` 再 super.dispose

伪代码：
```dart
final List<Timer> _initJumpTimers = [];

void _scheduleInitialJump() {
  for (final ms in const [0, 80, 200, 400]) {
    _initJumpTimers.add(Timer(Duration(milliseconds: ms), () {
      if (!mounted) return;
      _jumpToBottom();
    }));
  }
  _initJumpTimers.add(Timer(const Duration(milliseconds: 450), () {
    if (mounted) _initialScrollDone = true;
  }));
}

@override
void dispose() {
  for (final t in _initJumpTimers) t.cancel();
  _initJumpTimers.clear();
  _scrollCtrl.removeListener(_onScroll);
  _scrollCtrl.dispose();
  super.dispose();
}
```

额外的防御层（不改 timer 机制，仅在 `_jumpToBottom` 内多一道兜底）：
```dart
void _jumpToBottom() {
  if (!_scrollCtrl.hasClients) return;
  final pos = _scrollCtrl.position;
  if (!pos.hasContentDimensions) return;  // RenderObject 已 detach 时 hasContentDimensions = false
  _scrollCtrl.jumpTo(pos.maxScrollExtent);
}
```

两层防御都加，确保万无一失。

## 4. 影响范围

- **改动文件**：`AndroidClient/lib/features/chat/widgets/message_card_list.dart` 一份文件
- **行数**：增加 ~10 行（timer 列表 + cancel 循环 + hasContentDimensions 检查），无删除
- **行为变化**：
  - 进入聊天页正常流程：完全无变化（timer 照常 fire → jump 照常执行）
  - 快速返回场景：timer 被 cancel，永远不会进入 _jumpToBottom，崩溃消除
  - 进入后 ListView 还未 layout 的极端帧：hasContentDimensions check 拦截，安全降级为 noop
- **不影响**：Mac 端、Server、其他 Android 页面
- **风险**：极低。改动局限在一个 widget 的内部 timer 管理，无任何业务逻辑变化

## 5. 验证方式

| 场景 | 步骤 | 期望 |
|---|---|---|
| 正常进入工作区滚到底 | 进入有历史消息的工作区 | ListView 自动滚到最新消息位置 |
| 快速进出工作区 | 进入空工作区立即点返回 | 平滑回主页，无红屏，logcat 无 _owner assertion |
| 慢速进出工作区 | 进入工作区等 ~1s 再返回 | 同样无报错，正常行为 |
| 反复进出多次 | 连续进出 10 次 | 无累积内存泄漏 / 卡顿 |
| Mac 历史加载途中返回 | 进入大对话历史的工作区，未加载完前返回 | 无报错 |
