## 跨需求耦合点(本轮重点)

### #20(@<path> 改写)+ #28(attachment dedup)+ #15(tool_use status 升级)在同一条 user → tool_use Read → tool_result 流上

模拟流程:
1. Phone 发 attachment(local-uuid A)→ Mac → upload → JSONL 写 user message(server-uuid X)`@/Users/.../inbox/IMG.jpg`
2. JSONL 进来 → `_rewriteInboxRefAsAttachment` 把 user X 改写为 attachment 卡片
3. _mergeMessages attachment dedup 命中 local-A,uuid 升级为 X,保留 localPath
4. Claude 触发 tool_use(Read 该图) → JSONL 写 assistant tool_use 行(server-uuid Y)
5. Claude 读完 → JSONL 写 user tool_result 行(server-uuid Z,tool_use_id=Y)
6. _mergeMessages 扫描 toolResult 把 Y 的 toolStatus 升级为 executed

**链路完整,无副作用**。OK。

### #23(text dedup)+ #20(text → attachment rewrite)同时存在时的处理顺序

模拟:用户发 user text "看下这个 @/Users/.../inbox/IMG.jpg" (混合 text + path)
- `_rewriteInboxRefAsAttachment` 的 regex 要求 **整条 text 是 `@<path>` 形式**(`^@(...)\s*$`)→ 混合内容不命中,继续走 text dedup
- _consumeMatchingPendingText 按完整 text 匹配,本地 echo 是 "看下这个 @..." → 应能匹配(只要 sendText 时把 path 这种字符串完整发了)
- ✅ 不冲突

但是边界:用户先单独发 user text "看下这个",再单独发 attachment → 两条独立消息,各走各的 dedup,无问题。

### `permission-mode / last-prompt / bridge-session` 兜底过滤 + `<command-name>` 过滤的相互影响

`<command-name>` 是 string content 分支(line 181-184),`bridge-session/permission-mode` 是 topType 顶层分支(line 207-217)。兜底 `topType != null && raw['message'] == null && raw['sessionId'] != null`(line 219)放在最后,只 catch 不带 message 字段的元数据,**不会误吃 string content user message**(因为那时 inner.content 是 string,raw.message 非 null)。✅ 链路 OK。

### #16/#17(image 协议)与 #28/#31(dedup)的协同

- attachment 从 JSONL 进来 → 走 attachment dedup → 同时触发 `_requestImageDownloadUrl`
- 这里有时序:dedup 把 local-A 替换为 server-X,然后 `for (final m in items)` line 524 用的是 items 中的 m(server-X 的 message),所以 `_pendingDownloadUrl` 注册的 msgUuid 是 server-X
- response 回来时 indexWhere(x.uuid == server-X) 命中刚 dedup 完的位置 ✅
- 但**如果 dedup 失败(local-A 已被另一条 server message 占用 / 同名图问题 #1)** → m 的 server uuid 没找到对应的本地 echo,新插入一条 server attachment,触发 download.url 请求 → response 回来命中 server-X 位置(刚插入的),OK。

但叠加 🔴 #1:同名图导致 ImageRefStore 第一张的 upload_id 被覆盖 → `_requestImageDownloadUrl(filename)` 查 store 得到的是**第二张的 upload_id**,response 回来填到**第一张卡片**(uuid_X1) → **错位**:第一张卡片显示第二张图。
