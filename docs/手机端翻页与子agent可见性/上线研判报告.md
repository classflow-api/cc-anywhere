## 6. 最终结论

### 🛑 必须先完成人工测试才能上线

**理由**：

1. **维度 4 = 🔴**：跨端 + UI 手势 + 实时通信 + 文件系统监听 + 协议向下兼容的全栈改动，**完全无自动化测试基建兜底**，依赖代码 review 的逻辑正确性证明已不足以独立替代真机回归。

2. **维度 2 + 3 = 🟡×2 叠加**：虽然单维度未到 🔴，但组合效应明显 — "无测试 × 高风险 hot path 改动 × 双端 7 字段协议演进" 形成放大效应，需要人工补足覆盖。

3. **R-F7-001 / 实测 4 至今未跑**：JSONL实测结果.md 明示 permission mode 继承"按推理默认成立"，三轮 review 均建议"上线前补一次集成测试"。**这是直接 P0 阻塞项** — 如果 acceptEdits 模式下子 agent 仍触发卡片，用户日常工作流（自动批准 Edit）会被打断。

4. **三轮 review 全通过 = 🟢 但有 6 项 🟢 信息项保留**：6 项 🟢 中至少 2 项（🟢-2 漏 chip / 🟢-5 视觉冗余）属于"实测中可能可见的小毛病"，需要 dogfood 实地观察确认用户可接受。

5. **改动量大 = 强 review 已闭环但生产环境差异未覆盖**：本次涉及 25 文件 +2138 行（剔除 L3 后仍有 ~2000 行 L4 范围），三轮 review 在代码逻辑/边界/质量三维度均通过且复审独立验证，但 SwiftTerm/FSEvents/Flutter Widget 渲染等**与 Apple/Google 系统 API 交互的行为**只有在生产 .app/.apk 上才能完整暴露。

### 综合判定

- ✅ 三轮 review 全闭环（**代码正确性已穷尽 paper-review 能力**）
- ❌ 无自动化测试 + 关键集成点（R-F7）未实测
- → **必须先完成 §5 P0 场景（V1-V3 / V6-V9 / V13 / V15）人工验证后方可上线**
- → P1 / P2 场景可在上线后 dogfood 期间持续验证

## 7. 上线后建议

### 7.1 验证流程

1. **上线前必跑**：§5.1-§5.5 全部 P0 场景（V1-V3 / V6-V9 / V13 / V15）— 约 1 小时
2. **上线第一天**：dogfood 1 天观察日志（重点看 Mac line 244 GC / line 606 反查命中率 / line 608 反查未命中频率）
3. **上线第一周**：先内部 dogfood 1 周再开源（个人工具，可先在自己环境用足 7 天）

### 7.2 监控点

- Mac 端 `subagent gc: dropped tasks=N agents=N` 是否经常出现 droppedAgents > 0（暗示 promptHash 匹配失败常态化）
- Mac 端 `ask no sub-agent context` 在已知有 subagent 活跃时输出频率（暗示 race window 是否比 29ms 大）
- 手机端折叠块是否长期"运行中"未变 ✅（暗示父 tool_result 未到 → 检查 FSEvents 监听稳定性）

### 7.3 回滚预案

- 单 commit 回滚整段（git revert）— 三轮 review 已确认改动集中、回滚安全
- 协议字段全 optional 已提供天然向下兼容 — 即使只回 mac 不回手机 / 反之，**最差降级为现有体验**，不会出错

### 7.4 跟进项（不阻塞上线）

| 项 | 触发条件 | 处理流程 |

| 🟢-3 children 懒加载 | 产品反馈"长 Task 卡顿" | L1/L2 单独优化 |
