# AskUserQuestion 手机端实时交互 - 可行性评估

> 日期:2026-05-15
> 评估对象:cc-anywhere 是否能支持 phone 端实时参与 AskUserQuestion 决策
> 结论:**完全可行,推荐方案 B(Hook 阻塞中继),工程量 3-4 人日**

---

## 1. 问题本质

Claude Code Agent SDK 文档明确说明([source](https://code.claude.com/docs/en/agent-sdk/user-input)):

> Claude requests user input in two situations: when it needs **permission to use a tool**, and when it has **clarifying questions** (via the `AskUserQuestion` tool). Both trigger your `canUseTool` callback, which **pauses execution until you return a response**.

也就是:

- AskUserQuestion 是 SDK 层的**阻塞回调**,**未答完前 JSONL 不写**
- 答完后,`tool_use` + `tool_result(含 answers)` **一次性 flush 到 JSONL**

cc-anywhere 当前"JSONL 旁观者"架构,**架构上不可能** 看到 pending 的 AskUserQuestion。

---

## 2. 两条架构修复路线对比

### 方案 A:用 Claude Agent SDK 包装 Claude CLI

把 Mac 端 cc-anywhere 从"监控 jsonl"改为"用 SDK 启动 claude 子进程",在 `canUseTool` 回调里拦截 AskUserQuestion → ws 推 phone → phone 回 → resolve 回调。

| 维度 | 评估 |
|---|---|
| 改动量 | 5-8 人日(完全重构 Mac 端 ProcessHost + JSONLWatcher 等) |
| Mac TUI 体验 | ❌ **彻底失去** — 用户不再直接键盘 talk 给 claude,而是 cc-anywhere 控制的子进程,SwiftTerm 失去意义 |
| 优点 | SDK 是 first-class API,接口稳定 |
| 缺点 | 跟项目"无 daemon,用户直接 typing claude TUI"的核心定位冲突 |
| **结论** | ❌ **不推荐** |

---

### 方案 B:PreToolUse hook 阻塞中继 ⭐ 推荐

利用 Claude Code 现有的 [PreToolUse hook 机制](https://code.claude.com/docs/en/agent-sdk/hooks):

1. `~/.claude/settings.json` 注册一个 PreToolUse shell command hook,`matcher: "AskUserQuestion"`
2. Hook 是一个简单脚本,**同步阻塞**直到 phone 答完
3. Hook 输出 `{hookSpecificOutput: {permissionDecision: "allow", updatedInput: {questions, answers}}}` 给 SDK,SDK 跳过原本的 TUI 弹问题,直接拿这个 answer 继续

**关键文档证据**:
- Hooks 是同步阻塞的:"the agent waits for your hook to return before proceeding"
- Hook 可设 timeout(默认 60s,可调更长)
- Hook **同时支持 SDK 和 CLI** — 项目里 yoolines-dev-workflow / superpowers / episodic-memory 这些插件都通过 settings.json hooks 工作,**已经被用户在用**,证明 Claude CLI TUI 模式会执行这些 hooks
- Hook 可以 `updatedInput` 直接返回 `{questions, answers}`,完全绕过 TUI 内置弹窗

| 维度 | 评估 |
|---|---|
| 改动量 | **3-4 人日** |
| Mac TUI 体验 | ✅ **完全保留** — Mac 端用户直接 typing,只是 AskUserQuestion 弹窗不会出现在 TUI,被 hook 透传到手机 |
| 优点 | Claude 官方机制,无需重构现有架构,跟 cc-anywhere 定位完全契合 |
| 缺点 | hook timeout 限制(手机端答题超时需处理),需要 IPC 通道(hook 脚本 ↔ Mac App) |
| **结论** | ✅ **推荐** |

---

## 3. 方案 B 详细实施拆解

### 组件 1:Hook 脚本(`hook-ask-user-question.py`)

- 位置:`~/.local/bin/cc-anywhere-hook-ask` 或随 cc-anywhere Mac App 分发
- 入参:stdin 读 JSON(`{hook_event_name, tool_name, tool_input: {questions: [...]}, session_id, cwd}`)
- 流程:
  1. 通过 Unix socket(`~/Library/Application Support/cc-anywhere/hook.sock`)POST 给 Mac App
  2. 阻塞等 Mac App 返回 answer JSON(或 timeout/cancel)
  3. 输出 `{hookSpecificOutput: {permissionDecision: "allow", updatedInput: {questions: [...], answers: {...}}}}` 到 stdout
- 工程量:~80 行,**0.5 人日**

### 组件 2:Mac App IPC + ws 桥接

- 在 cc-anywhere Mac App 启动一个 Unix socket server(`hook.sock`)
- 收到 hook 请求 → 生成 request_id → 通过 ws 推 phone:`ask.question.pending {request_id, tab_id, questions}`
- 等待 phone 通过 `ask.question.answer {request_id, answers}` 回复
- 把 answers 通过 socket 写回 hook 进程
- 工程量:Mac Swift IPC server 100 行 + protocol 2 个 type,**1 人日**

### 组件 3:Server 协议透传

- `ask.question.pending`(mac→server→phone)+ `ask.question.answer`(phone→server→mac)
- 路由层加 2 个 case
- 工程量:**0.5 人日**

### 组件 4:Phone 端 UI 适配

- 现有 `AskUserQuestionCard` 已经渲染 questions + options(我之前实现的,从 JSONL 读)
- 改造:接收 `ask.question.pending` 后**立即弹卡片**(不依赖 JSONL),用户选完发 `ask.question.answer`
- 工程量:**0.5 人日**(主要是消息路由 + 状态管理)

### 组件 5:自动安装

- Mac App 启动时检查 `~/.claude/settings.json` 是否含 cc-anywhere hook,缺则提示/自动写入
- 一份 markdown 教程让用户能选择手动安装
- 工程量:**0.5 人日**

**总计:3-4 人日**

## 4. 风险点与对策

### 风险 1:Hook 超时

- 默认 60s,远短于"用户离开手机一会再回来"的真实场景
- 对策:在 settings.json 配置 `timeout: 1800`(30 分钟)
- 备用对策:hook 用 [`defer` 决策](https://code.claude.com/docs/en/hooks#defer-a-tool-call-for-later)持久化 session,后续从外部 resume

### 风险 2:Phone 端无响应(网络断 / 用户长时间不操作)

- Mac App 设 timeout(比如 5 分钟),超时后 hook 返回 `{permissionDecision: "deny", permissionDecisionReason: "phone timeout"}` 让 Claude 自行处理
- Claude 收到 deny 会自动选个合理默认或换思路

### 风险 3:Mac TUI 端用户也想在本地答题(双向兼容)

- 方案 1:hook 阻塞期间 Mac App 也展示 phone 看到的同一卡片(在 Mac App UI 里,不在 Claude TUI 里),Mac 端用户可点
- 方案 2:hook 配置选项 `prefer_remote: true/false`,prefer_remote 时阻塞;false 时直接 deny 让 TUI 处理

### 风险 4:多 phone 接收时的竞态

- 同时多 phone 都看到问题 → 第一个回答的 wins,其他 phone 显示"已被 X 回复"
- 通过 request_id + Mac App 端单 winner 锁实现

### 风险 5:Hook 安装/卸载干扰用户其他 hooks

- 严格按 settings.json 现有 PreToolUse 数组 append,不覆盖
- 卸载时只删 cc-anywhere 自己加的那一项
- Mac App 偏好里加开关,默认开启

## 5. 验证路径(开工前最小风险验证,30 分钟)

最大未知风险是 "Claude CLI TUI 模式是否真触发 settings.json 的 PreToolUse hook"。验证步骤:

1. 写一个最简单的 hook 脚本:
   ```bash
   #!/bin/bash
   read -r line
   echo "$(date) $line" >> /tmp/cc-hook-test.log
   echo '{}'
   ```
2. 在 `~/.claude/settings.json` 加:
   ```json
   {
     "hooks": {
       "PreToolUse": [
         { "matcher": "AskUserQuestion",
           "hooks": [{ "type": "command", "command": "/path/to/script.sh" }] }
       ]
     }
   }
   ```
3. 在 Mac TUI 里让 Claude 触发 AskUserQuestion(写 prompt 让它问澄清问题)
4. 检查 `/tmp/cc-hook-test.log`

如果日志写入 → 路径完全打通,可以开干
如果没写入 → 需要进一步研究 CLI 跟 SDK 的 hook 加载差异

## 6. 推荐决策

**建议**:

1. **今天先做 30 分钟验证**(上面 §5),确认路径打通
2. 验证通过后,**作为下个迭代的 L4 中型需求**正式开发(3-4 人日)
3. 当前阶段保留我已实现的 phone 端 AskUserQuestionCard(从 JSONL 解析,只读)— 让用户**事后**能看到决策记录,有信息价值

**关键收益**:

- 方案 B 是 cc-anywhere "远程协作" 定位的**最佳契合**实现
- 完成后,phone 端真正能成为"决策端",Mac 端只是 claude 进程的"承载"
- 同样的 hook 机制可扩展到所有 `PreToolUse` 场景(其他需要批准的工具如 Bash/Write),**一举多得**
- 这是 cc-anywhere 区别于 cc-connect 的核心差异化能力(cc-connect 是 IM 通知,cc-anywhere 是真·远程交互)
