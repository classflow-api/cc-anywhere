# Mac 端偏好设置输入框无法编辑 - 修复方案

| 文档信息 | 值 |
|---|---|
| 版本 | v1.0 |
| 编写日期 | 2026-05-15 |
| 缺陷级别 | L2 - Bug 修复 |
| 影响端 | MacClient |

## 1. 问题描述

### 问题现象
- 用户在 Mac 端「偏好设置 → Server 连接」面板中点击「Server 地址 / 端口 / 主 Token」三个输入框,无法输入文字。
- 进一步验证发现:整个偏好设置窗口、主窗口里的所有 SwiftUI `TextField`/`SecureField` **都不响应键盘**(不仅是 Server 面板)。
- 表象只是"输入框点不进字",但根本是窗口拿不到键盘焦点(无法成为 key window)。

### 期望行为
- 点击 Server 地址输入框后,光标进入输入框,键盘输入立即在输入框中显示。
- 输入完毕后,变更通过 `@Published serverConfig` 的 `didSet` 持久化到 `~/Library/Application Support/cc-anywhere/server-config.json`。
- 其他偏好设置面板的输入框(GeneralPane 的 claude 路径覆盖)同样可正常输入。

### 复现步骤
1. 在 MacClient 目录下执行 `swift build`
2. 直接命令行运行 `.build/debug/CCAnywhere`(README 中提到的"直接运行 binary"方式)
3. App 启动后,通过 `Cmd+,` 或菜单打开「偏好设置」
4. 切到「Server 连接」面板
5. 点击「Server 地址」输入框尝试键入字符 → 无任何反应

## 2. 根因分析

### 触发链路
1. `.build/debug/CCAnywhere` 是 SPM 编译产物 — **裸 Mach-O binary,不是 `.app` Bundle**。
2. Launch Services 启动一个进程时,通过 Bundle 的 `Info.plist` 决定 ActivationPolicy(Regular / Accessory / Prohibited)。
3. 裸 binary **没有 Bundle Info.plist**(`CCAnywhere/Sources/CCAnywhere/Resources/Info.plist` 存在,但只有打成 .app 才会被 Launch Services 读到)。
4. 命令行启动时,默认 ActivationPolicy 是 `.accessory` 或 `.prohibited` → **App 被注册成 background-only 进程**。

### 现象验证
通过 `System Events` 查询当前 CCAnywhere 进程状态:
```
background only: true   ← 系统认为是后台进程
visible: false          ← 不在 Dock / Mission Control 可见
frontmost: false        ← 永远拿不到 frontmost
```

### 为何输入框失效
- Background-only 进程的 SwiftUI `WindowGroup` 仍会创建 NSWindow 并渲染,**但窗口无法成为 key window**。
- macOS 的键盘事件 dispatch 链:`NSApp` → key window → first responder → `NSTextField/SwiftUI TextField`。
- key window 不存在,键盘事件根本没机会到达 `TextField` 的 `becomeFirstResponder` / `keyDown:`。
- 所以**所有输入框无差别失灵**,与 ServerPane / PreferencesService / ServerConfig 的 binding 实现没有任何关系。

### 二次验证(代码层面排除)
- `PreferencesService.serverConfig` 是 `@Published var serverConfig: ServerConfig`,`ServerConfig` 是 struct
- `ServerPane` 中绑定 `$preferences.serverConfig.server` / `.port` / `.masterToken` 链路全部合法
- 全工程 `grep -rn "disabled\|allowsHitTesting\|focusable.*false"` 无匹配
- 因此**代码 binding 没问题,bug 在运行时窗口状态层面**

### 旁证(README 已暗示)
项目 `MacClient/README.md` "构建" 章节写:
> 直接运行该文件即可在终端中启动 App(注意:作为命令行启动时,**没有 Info.plist bundle,部分 macOS 特性可能受限**)。

"部分 macOS 特性可能受限"就包含 ActivationPolicy / 键盘 focus 行为。README 推荐用 Xcode 启动以获得完整 App bundle,但即使 Xcode 启动也会构造一个临时 bundle —— 这是约定俗成的解法,不解决"裸 binary 启动"路径。

## 3. 修复方案

### 修改思路
在 `AppDelegate.applicationDidFinishLaunching` 中**主动调用** `NSApp.setActivationPolicy(.regular)`,显式声明 App 是 Regular GUI App(显示 Dock 图标、可前台、可成为 key window)。这样无论从 .app Bundle 启动还是裸 binary 启动,行为都一致。

### 具体修改

**文件**: `MacClient/Sources/CCAnywhere/App/AppDelegate.swift`

`applicationDidFinishLaunching` 函数最前面新增 2 行:

```swift
public func applicationDidFinishLaunching(_ notification: Notification) {
    // 命令行直接运行裸 binary 时,Launch Services 默认把进程注册为 .accessory(背景进程),
    // 导致 SwiftUI 窗口可见但无法成为 key window,所有 TextField 都收不到键盘事件。
    // 显式声明 Regular,确保无论从 .app Bundle 还是裸 binary 启动行为一致。
    NSApp.setActivationPolicy(.regular)
    NSApp.activate(ignoringOtherApps: true)

    container?.appDidFinishLaunching()
    // ...其余不变
}
```

### 修改前后对比

| 维度 | 修改前 | 修改后 |
|---|---|---|
| .app Bundle 启动 | Regular(Info.plist 默认) | Regular(显式) |
| 裸 binary 启动 | Accessory / Prohibited | **Regular** |
| Dock 图标 | 裸 binary 启动时不显示 | **始终显示** |
| Cmd+Tab 可切到 | 裸 binary 启动时不可 | **始终可** |
| TextField 键盘输入 | 裸 binary 启动时失灵 | **始终可用** |

## 4. 影响范围

### 修改文件
- `MacClient/Sources/CCAnywhere/App/AppDelegate.swift`(单文件,+5 行含注释)

### 行为影响
- **正向**:所有键盘焦点 / Dock 显示相关功能在裸 binary 启动下也能正常工作
- **不破坏现有 .app Bundle 启动行为**:已是 Regular 的进程,再调一次 `setActivationPolicy(.regular)` 是幂等的
- **不引入新依赖 / 不修改架构**:只是一行 NSApp 调用

### 潜在副作用评估
| 副作用 | 评估 |
|---|---|
| 影响主窗口 SwiftUI WindowGroup | 否。setActivationPolicy 在 willFinish/didFinishLaunching 调用是 AppKit 推荐时机,WindowGroup 在此之后才请求窗口 |
| 影响菜单栏注册 | 否。AppDelegate 的 `installMainMenu` 在此之后运行,菜单不变 |
| 影响 background mode 需求 | 项目 README 明确"无 Mac 端 daemon",cc-anywhere 永远是 GUI App,不存在希望 background-only 启动的场景 |
| 影响测试 / CI | 否。Swift Package 的 `swift test` 不会经过 AppDelegate |

## 5. 验证方式

### 主路径
1. `swift build` 重新构建
2. 杀掉已运行的 CCAnywhere(`pkill CCAnywhere`)
3. 重新命令行启动 `.build/debug/CCAnywhere`
4. Dock 中应出现「遥指」/「CCAnywhere」图标
5. 主窗口拿到前台焦点
6. `Cmd+,` 打开偏好设置 → Server 面板 → 点击 Server 地址输入框 → 键入字符 → **可见字符出现,光标移动**
7. 关闭偏好设置窗口后,查 `~/Library/Application Support/cc-anywhere/server-config.json` → server 字段已更新

### 回归测试
- 「偏好设置 → 通用」面板 claude 路径输入框可输入(同 binding 机制)
- 「偏好设置 → 安全」面板若有输入框可输入
- 主窗口 ChromeBar 的可点击控件(按钮)正常响应
- Xcode 启动方式(开 `Package.swift` Run)的行为不退化:Dock 图标、菜单栏、键盘 focus 都正常

### 失败回滚
若修改后出现任何回归,删除新增的两行 `NSApp.setActivationPolicy(.regular)` / `NSApp.activate(ignoringOtherApps: true)` 即可完全回滚到原行为。

## 6. 关联讨论

### 不修代码 ServerPane 的理由
- ServerPane / ServerConfig / PreferencesService 的代码完全正确,binding 链合法
- 此 bug **不是 binding bug**,而是窗口焦点的 bug
- 若改 ServerPane 反而会引入无用的 workaround,且不能根治"所有 TextField 都不能输入"的真问题

### 为何不改 Info.plist
- Info.plist 已存在且配置正确(`LSApplicationCategoryType` 为 developer-tools,无 `LSUIElement`/`LSBackgroundOnly`)
- Info.plist 仅在打成 .app Bundle 时才生效,裸 binary 启动场景不受其影响
- 不需要修改 Info.plist
