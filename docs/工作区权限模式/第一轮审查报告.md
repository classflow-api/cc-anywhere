### ✅ 2. `Tab.swift` Codable 兼容性

`Tab.init(from:)` 第 74-79 行：

```swift
if let raw = try c.decodeIfPresent(String.self, forKey: .permissionMode),
   let m = PermissionMode(rawValue: raw) {
    self.permissionMode = m
} else {
    self.permissionMode = .default
}
```

- `decodeIfPresent` 处理"字段缺失"路径；
- 内层 `PermissionMode(rawValue: raw)` 处理"字段存在但值非法（如以前误写、降级后又升级）"；
- 两条路径都 fallback 到 `.default`，不抛错。
- 类型用 `String` 而非 `PermissionMode` 直接解码 —— 这是**正确选择**，否则一旦 JSON 里值非法（哪怕只是大小写错），整个 Tab 解码会 throw，导致 `loadFromDisk()` 把整个 `tabs.json` 全清空（catch 里设置 `tabs = []`）。**通过**。

老 json 数据（无 `permission_mode` 字段）反序列化：`decodeIfPresent` 返回 nil → 走 else → fallback `.default`。**不会抛错**。**通过**。

### ⚠️ 4. `TabManager.setPermissionMode`

第 73-87 行：

- ✅ id 不存在 → return nil
- ✅ mode 相同 → return nil（去抖）
- ⚠️ **持久化失败处理**：`persist()` throw 后**先 log，再 return nil**，但 `tabs[idx].permissionMode = mode` 这一行**已经把内存中的 mode 改了**。也就是说：持久化失败时，**磁盘是旧值，内存是新值，状态不一致**。如果此时调用方根据 nil 返回值 abort 重启 claude，下次 App 重启 mode 会回到旧值，但本次运行中 UI 会显示新 mode → 用户困惑。

  严格说这是个数据一致性缺陷。但鉴于：
  1. 个人工具语境，`persist()` 写入 `~/.appsupport` 失败极罕见；
  2. 调用方 SidebarView/TabStripView 拿到 nil 后直接 `return` 不重启，UI 也不会刷新（mode 显示走的是 `tab.permissionMode`，而 SwiftUI 视图绑定的是 `@Published tabs`，重新发布出去后 row 会刷新 → 但 popup 的 checkmark **会**变成新值，这就和磁盘脱钩了）。

  **建议**修复：失败时把 mode 回滚 `tabs[idx].permissionMode = oldMode` 再 return nil。
- ✅ `.renamed` 事件触发：从 TabSyncBridge.swift:73 可见 `.renamed` 转换为 `action = "renamed"` 的 `tab.changed` 广播。`TabSummary` 不含 permission_mode 字段，所以手机端只会收到一条 `tab.changed renamed`，name/folder/status 都没变 → 对手机端是 **no-op**，对齐"AndroidClient 零改动"。但**会产生一次无意义的 ws 推送**（小开销，可接受）。
- 标记为 🟡 建议，不阻塞。

```swift
let hasHistory = ...  // jsonl 文件检测
```

- ⚠️ 第 74 行 banner 文字：`view.feed(text: "\u{1B}[36m[cc-anywhere] launching: \(exe) \(mode) --permission-mode \(tab.permissionMode.rawValue)\nin: ...")` —— banner 中**始终**显示 `--permission-mode`，包括 hasHistory=false 路径，OK。但 banner 里的 `\(mode)` 是变量 `let mode = hasHistory ? - **通过**（核心 args 拼接正确，命名问题不阻塞）。

### ⚠️ 6. `TabUIHelpers.askPermissionMode`

第 39-58 行：

- ✅ `withExtendedLifetime(target) { alert.runModal() }`：`runModal()` 是同步阻塞调用，target 在 closure 内被强引用，期间 ARC 不会释放它 → popup `target = target` 的 weak/unowned 引用安全（NSPopUpButton.target 是 weak）。**正确**。
- ✅ 返回 nil 表示取消（`response != .alertFirstButtonReturn`）；选 default 时返回 `PermissionMode.default`（非 nil） → **可区分**。
- ⚠️ 第 57 行 `guard idx >= 0, idx < PermissionMode.allCases.count else { return defaultMode }` —— `indexOfSelectedItem` 在异常情况返回 -1，此时**返回 defaultMode 而不是 nil**，从语义上不太一致（"取消"返回 nil，"异常但用户点确定"返回 defaultMode）。但实际上 popup 选项填满 6 个，用户点了确定 idx 必然在 0...5，这个分支几乎不可能命中。**🟡 建议**改成 return nil 或 fatalError（理论上不可达）。
- ✅ NSPopUpButton 第 22 行第一次创建带尺寸，第 44 行又改 frame —— 这两次设置后者覆盖前者，最终生效的是第 44 行的 `(0, 40, 360, 26)`，没问题，但第 22 行的 frame 是死代码可优化。
- ✅ NSTextField summary：第 31-36 行设的 frame 也被第 45 行覆盖。同上，死代码，不影响功能。
- **通过**（功能正确，几处冗余代码留给 R3）。

### ⚠️ 8. changePermissionMode 逻辑

SidebarView 第 156-173 行 / TabStripView 第 103-118 行（基本完全一致）：

- ✅ `guard tab.permissionMode != mode else { return }` —— 同 mode 直接 return，不弹确认框。**正确**（避免无用对话）。
- ✅ NSAlert 确认 → 用户取消则 return。
- ✅ `setPermissionMode` 返回 nil 处理：`guard let updated = ... else { return }`。
- ⚠️ **0.2s 延迟重启的合理性**：注释说 "stopProcess 内部会同步停掉，但 PTY clean-up 是异步的，给一点 buffer 防止文件句柄竞争"。从 ProcessHost.stopProcess（第 92-99 行）来看，只是 `view.terminate()`（发 SIGTERM）+ `terminalsByTab.removeValue` —— 子进程退出确实是异步的，且 claude CLI 退出后还要刷新 `.claude/projects/.../session.jsonl` 等文件；如果立即 startProcess 用 `-c`，可能读到上一次的 session 文件锁状态、或者刚好两个 claude 实例重叠 1 秒（旧的还在写 jsonl，新的开始 watch）。

  0.2s 是经验值，不严谨但够用；个人工具够。**通过**，但留有微小竞态可能（旧子进程没在 200ms 内死掉时新 claude 会启动并去 `-c` —— claude 自己可能能处理，因为 session 文件由文件名隔离，且新进程拿到的是 fresh PTY）。
- ⚠️ **DispatchQueue.main.asyncAfter 捕获 `processHost`**：`processHost` 是 EnvironmentObject，在 SwiftUI struct view 里**值类型 view 重建后捕获的可能是旧实例的引用**。但 `ProcessHost` 是 `@MainActor final class : NSObject, ObservableObject`，被 `@EnvironmentObject` 注入是 class 实例（reference type），整个 App 生命周期单例 → 闭包捕获 self.processHost 指向同一个 ProcessHost 实例，**没问题**。
- **通过**。

  ```swift

      Text(m == tab.permissionMode ? "✓ " + m.displayName : "    " + m.displayName)
  ```

  或者用 macOS 14 起更稳定的 `Toggle`/`Picker` API。但本轮主题是正确性，**功能上不阻塞**，留给 R3。

## 发现的问题

### 🔴 阻塞（必须修才能进下一轮）

无。本轮核心逻辑正确。

### 🟡 建议（不阻塞）

1. **`TabManager.setPermissionMode` 持久化失败时内存与磁盘脱钩**（TabManager.swift:76-82）
   建议在 catch 分支回滚 `tabs[idx].permissionMode` 到旧值，再 return nil。

2. **`ProcessHost.startProcess` 局部变量 `mode` 命名歧义**（ProcessHost.swift:73）
   与 `tab.permissionMode` 概念混淆。建议重命名为 `historyHint` 或 `runMode`。

3. **`SidebarView.createTab` 多余的 `selectedTabId = tab.id`**（SidebarView.swift:144）
   `TabManager.createTab` 已经设过；删除避免误导。

4. **`TabUIHelpers.askPermissionMode` 异常 idx 路径**（TabUIHelpers.swift:57）
   `indexOfSelectedItem` 为 -1 时返回 `defaultMode` 与"取消返回 nil"的语义不一致。建议返回 nil 或加 `assertionFailure`。

5. **`TabUIHelpers.askPermissionMode` 死代码 frame**（TabUIHelpers.swift:22, 32-33）
   `popup` 和 `summary` 的 frame 第一次设置被第 44-45 行覆盖，可删除冗余 frame 赋值。

6. **`contextMenu Label + systemImage` 渲染不稳定**（SidebarView.swift:55, TabStripView.swift:31）
   macOS SwiftUI contextMenu 对 Label 图标渲染历史上不稳定。建议改为前缀字符 `"✓ "` 或在 R3 实测验证。

7. **设置 mode 触发 `.renamed` 事件 → 无意义 ws 推送**（TabManager.swift:85）
   `TabSummary` 不含 permission_mode，手机端收到的 tab.changed 实际无变化。功能上无害，但理论上可增加专属 `.permissionModeChanged` 事件（或在 sendTabChanged 中跳过 noop）。这是变更说明 §3.1 明确接受的"简单起见用 .renamed"，**不算缺陷，仅信息**。

### ⚪ 信息

1. **0.2s 重启延迟是经验值**（SidebarView.swift:170 / TabStripView.swift:115）
   不严谨，但个人工具语境可接受。若 R3 实测发现偶现"两个 claude 实例并发写 jsonl"，再升级到"等 processTerminated 回调再启"。

2. **`Tab.permissionMode` 默认初始化两次**（Tab.swift:28 默认值 + init 默认参数）
   Swift 允许，结构体 `permissionMode: PermissionMode = .default` 给 Codable 默认值兜底，init 的默认参数给手动构造兜底。两处一致都是 `.default`，OK。

3. **`PermissionMode.fallback` 常量未被使用**（PermissionMode.swift:43）
   `Tab.init(from:)` 中硬编码 `.default` 而非 `.fallback`。建议统一引用 `.fallback`，将来一处改全部生效。**🟡 微建议**。
