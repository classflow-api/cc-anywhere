## 检查清单结果（逐项）

### ✅ 1. 命名与可读性

#### PermissionMode rawValue 对齐

- `claude --permission-mode <mode>` 接受的值（claude --help 实测）：`acceptEdits / auto / bypassPermissions / default / dontAsk / plan` —— 全为 **camelCase**（首字母小写 + 后续单词首字母大写）
- Swift 枚举 case 名（PermissionMode.swift:10-16）：`default / acceptEdits / plan / auto / dontAsk / bypassPermissions` —— 同 camelCase
- Swift 隐式 rawValue = case 名 → **逐字匹配** ✅

Swift 社区对枚举 case 的命名习惯本来就是 lowerCamelCase（[Swift API Design Guidelines](https://www.swift.org/documentation/api-design-guidelines/)），与 claude CLI 习惯重合 → **不需要为对齐 CLI 而 override rawValue**，代码更干净。**通过**。

#### displayName / summary 文案

PermissionMode.swift:19-40：

- `displayName`：「default · 只读」「acceptEdits · 读 + 编辑文件」「bypassPermissions · 所有操作（隔离容器）」等。**rawValue + 短中文** 的复合格式 → 既让懂英文的开发者一眼对应 CLI，又让中文用户能理解含义。**良好实践**。
- `summary`：「入门、敏感工作。每个工具调用都需要你逐项确认。」「仅适合隔离容器 / VM。绕过所有权限检查，慎用。」—— 句式一致（"场景。约束。"），措辞精准（如 bypassPermissions 显式写了"慎用"），**用户角度可理解**。

⚪ 唯一可挑剔：`dontAsk · 仅预批准的工具` 这一项含义稍抽象，但官方文档同样模糊（`Pre-approved tools only`），保留原文翻译已经是最佳做法。

**通过**。

### 🟡 4. 错误处理

#### setPermissionMode 返回 nil 时 UI 静默

`SidebarView.swift:163` / `TabStripView.swift:109`：
```swift
guard let updated = tabManager.setPermissionMode(tab.id, mode) else { return }
```

setPermissionMode 返回 nil 的 3 个原因：
1. id 不存在 — 不可能（菜单挂在该 tab 的 contextMenu 上）
2. mode 相同 — 已被 `guard tab.permissionMode != mode else { return }`（SidebarView:155 / TabStripView:101）提前拦截
3. **persist 失败** — 真实失败路径

**问题**：persist 失败 = "磁盘满 / I/O 错 / 权限错"等罕见但严重的情况。当前是**完全静默**：
- 日志有：`log.error("persist failed after setPermissionMode: \(error)")`（TabManager.swift:80）
- UI 无任何反馈 → 用户连点几次菜单都"没反应"，不知道为什么
- 而且 R1 已经发现：persist 失败时 **`tabs[idx].permissionMode = mode` 已经改了内存**，磁盘是旧值。这次提级到日志可见，但用户看不到。

**🟡 建议**（不阻塞，可在下次小迭代）：persist 失败时 `setPermissionMode` 内部回滚内存（`tabs[idx].permissionMode = oldMode`），并在调用方 catch 路径上加 `NSAlert("切换失败：persist error")`。这是 R1 建议 1 + 本轮加权。

#### loadFromDisk 两级降级

TabManager.swift:180-209：
1. 优先 `JSONDecoder.iso.decode([Tab].self, from: data)` 整体快路径
2. 失败则 `JSONSerialization.jsonObject` 得 `[Any]`，逐元素 reencode → decode Tab
3. 二级也失败 → `tabs = []`

**评估**：
- ✅ 解决了 R2 🟡 1（"permission_mode 字段类型错（非 String）致整个 tabs.json 解码失败 → 用户所有工作区列表丢"）
- ✅ 单 row 损坏会 `log.warn("dropping malformed tab row: \(raw)")`，可观测
- ✅ 整体 / 单 row / 全坏 三层日志（info / warn / error）齐
- ⚪ 唯一可挑剔：第 195 行 `try? JSONSerialization.data(withJSONObject: raw)` —— 如果 raw 是基础类型（不是 dict / array），data 序列化会失败被吞，row 直接丢；但实际 tabs.json 每行就是 dict，不可能命中

**通过**（R2 的中危问题已解决）。

### ✅ 6. 安全性

#### Shell Injection 风险评估

permissionMode rawValue 是**枚举 case 名**（不是任意用户字符串），**不能由用户自由输入**：
- 创建时来自 popup 6 选 1（TabUIHelpers）
- 修改时来自 contextMenu 6 选 1（SidebarView / TabStripView）
- 持久化反序列化时通过 `PermissionMode(rawValue: raw)` 严格校验，非法值 → fallback default

即便假设有人改 tabs.json 把 permissionMode 写成 `; rm -rf /` 这种字符串：
- 反序列化时 `PermissionMode(rawValue: "; rm -rf /")` = nil → 回 default ✅
- 假设它是合法 case 名（不可能），那再传给：

`SwiftTerm.LocalProcessTerminalView.startProcess(args: [String])` —— 通过 `posix_spawnattr / forkpty + execve` 启动（SwiftTerm/LocalProcess.swift:426 `startProcessWithForkpty`），**直接传 argv 数组到内核 syscall**，**完全不经过 shell 解析**。

→ **无 shell injection 风险** ✅

#### 持久化文件权限

`PreferencesService.atomicWrite(to:permissions:)`（PreferencesService.swift:174-184）固定写 `0o600`（owner rw, group/other 无权限）：
- `tabs.json` 用 0o600（TabManager.swift:214）✅
- 即便 permissionMode 落盘也只能本用户读 / 写

**通过**。
