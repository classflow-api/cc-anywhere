## 阶段一头脑风暴产出摘要

### 已确认的方案要点

**端架构**
- MacClient：Swift 原生 + SwiftTerm + 系统 PTY；多 Tab UI；每 Tab 关联文件夹，启动 `claude -c` 恢复
- AndroidClient：Flutter + 卡片式消息列表；支持文字+图片输入；tool_use 仅"批准/拒绝/总是批准"
- Server：Go + WebSocket over TLS（Docker + docker-compose 部署）；无状态消息路由

**核心原则**
- 无 daemon；软件退出 = 杀子进程 + 断连
- 对话持久化交给 Claude Code（`claude -c` + JSONL）
- 双通道：A(PTY → SwiftTerm) + B(FSEvents 监听 JSONL → 推手机)
- 个人工具：仅 TLS，无 E2E 加密，无多租户

**11 个细节决策**

| 决策点 | 选择 |
|--------|------|
| A. 消息缓冲策略 | A1 完全不缓冲，手机重连后拉 JSONL 补齐 |
| B. 历史加载范围 | B1 首次最近 50 条，向上滚动按需加载 |
| C. 推送频率 | C1 100ms 节流批处理，按消息 ID 去重 |
| D. tool_use 并发 | D1 不做控制，全塞 PTY 自然顺序处理 |
| E. 多手机输入冲突 | E1 与 D1 一致，全塞 PTY |
| F. Server 部署 | F2 Docker + docker-compose |
| G. macOS 最低版本 | G1 macOS 14+ (Sonoma) |
| H. Android 最低版本 | H1 Android 10+ (API 29) |
| I. Token 策略 | I1 主 token / sub_token 都不过期，可手动重置/撤销 |
| J. 日志方案 | J1 三端各自本地日志，UI 提供查看入口 |
| K. 错误可见性 | K1 顶部状态灯 + Tab 内错误提示 + Toast |
