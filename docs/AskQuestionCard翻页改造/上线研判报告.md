## 2. 改动概览

本次是 **L3 单文件 SwiftUI UI 改造**：将 `UserQuestionBody` 中"多问题纵向堆叠"的渲染方式改为"单题翻页"。当 Claude `AskUserQuestion` 工具一次提多个问题（典型 3-5 题）时，旧 UI 卡片高度可达 1000+ px，本次改为一次只渲染当前题，通过"上一题/下一题"按钮 + 键盘 ← / → / Enter 切换。

**改动核心**：

- 新增 `@State currentIndex` + `@FocusState keyboardFocused`
- 新增 `progressIndicator`（顶部 X/N 进度指示，仅 N≥2 显示）/ `navigationBar`（底部步进按钮）/ `submitButton`（抽出公共子视图）
- 新增 `.onKeyPress(.leftArrow / .rightArrow / .return)` 键盘交互
- 新增 `.animation(.easeInOut(duration: 0.15), value: currentIndex)` 局部翻页过渡

**严格保持不变**：

- 协议层（`ProtocolMessage.swift` / `AskCardRequestData` / `AskQuestionItem`）
- `AskQuestionCardController.submitUserQuestion(requestId:answers:)` 入口签名
- `HookIpcServer` winner 锁 / `resolveAsk` / `reapExpired`
- hook bridge Python
- 手机端 `ask_user_question_card_realtime.dart`
- 同文件 `ToolApprovalBody` 工具批准卡片
- `selections` / `otherTexts` / `canSubmit` / `submit()` 全部逻辑
- 200 字符截断、`otherSentinel` 渲染、答案聚合顺序等业务规则

## 6. 最终结论

### ⚠️ 可上线但建议补充人工测试

**详细理由**：

1. **三轮 Review 全部通过、无 🔴/🟡 遗留**：第一轮发现的 Enter 提交快捷键缺失（🔴）与死分支 onChange（🟡）已经独立验证修复（处置 A），第二、第三轮各自只有 🟢 信息项（pre-existing / 跨模块 / 文档措辞）。
2. **未触动高风险区域**：双通道、winner 锁、协议层、Controller 入口、hook bridge、手机端、ToolApprovalBody 全部经 grep 验证零改动。变更说明 §3.2 明确边界与第二轮 D 区 grep 结论一致。
3. **编译已验证通过**：主 Agent 执行 `swift build` 与 `bash build_app.sh release` 均退出 0（工作流元信息记录）。
4. **回滚方案极简**：单文件、纯 UI、零脏数据风险。
5. **但本次是 UI 改造，且 L3 未跑自动化测试**：变更说明 §4.1 列出 7 个验证场景全部需要实际启动 `.app` 触发 Claude `AskUserQuestion` 工具来人工冒烟（维度 4 🟡 + 维度 2 🟡）。其中关键交互（键盘 ← / → / Enter、@FocusState 与 TextField 焦点协作、`.onKeyPress` 与中文输入法、winner 锁打断翻页中的卡片）**只能人工验证**，代码 review 无法替代。

**因此**：可以上线，但**强烈建议**上线前由开发者本人或测试人员完成上述 §4.2 中 7 个场景的人工冒烟，重点确认：

- 场景 5（键盘翻页与 Enter 提交）— 三轮审查中风险最集中的部分
- 场景 6（自定义输入框文本翻页保留 + 中文输入法）— SwiftUI 焦点协作的复杂点
- 场景 7（winner 锁打断翻页中的卡片）— 与 winner 锁的解耦正确性
