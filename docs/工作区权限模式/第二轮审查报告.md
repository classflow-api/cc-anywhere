## 检查清单结果（逐项）

### ✅ 1. 持久化兼容性 — 旧 tabs.json 无 `permission_mode` 字段

**测试矩阵**（基于 Tab.init(from:) Tab.swift:66-81 + PermissionMode rawValue 表）：

| JSON 输入 | decodeIfPresent 行为 | PermissionMode(rawValue:) | 结果 |
|---|---|---|---|
| 字段缺失 | 返回 nil | 不执行 | → `.default` ✅ |
| `"permission_mode": "default"` | "default" | `.default` | → `.default` ✅ |
| `"permission_mode": "acceptEdits"` | "acceptEdits" | `.acceptEdits` | → `.acceptEdits` ✅ |
| `"permission_mode": "foobar"` | "foobar" | nil（非法 rawValue） | → `.default` ✅ |
| `"permission_mode": "Default"` 大写 | "Default" | nil（rawValue 大小写敏感） | → `.default` ✅（容错 OK） |
| `"permission_mode": ""` 空字符串 | "" | nil（PermissionMode 无空 case） | → `.default` ✅ |
| `"permission_mode": null` | nil（decodeIfPresent 对 null 也返回 nil） | 不执行 | → `.default` ✅ |
| `"permission_mode": 123` 类型不对 | **throws DecodingError.typeMismatch** | — | → 整个 Tab decode 失败 ⚠️ |

**关键发现**：除"类型不对（非 String）"外，所有路径都软失败回 `.default`，不抛错。**这一点优于直接 `decode(PermissionMode.self, forKey:)`**（R1 已认可）。

**「类型不对」路径分析**：
- Tab decode throws → 在 `JSONDecoder.iso.decode([Tab].self, from:)`（TabManager.swift:178）中冒泡
- `JSONDecoder` 默认行为：**数组里任一元素 decode 失败 → 整个数组 decode 失败**（没有 lossy decoding）
- loadFromDisk catch（TabManager.swift:182-185）：把 `self.tabs = []`
- **后果**：用户**所有 Tab 列表丢失**！只因为某个 Tab 的 `permission_mode` 被某种方式写成了 number/boolean/array

**风险评估**：
- 实际触发概率极低：JSON 只由本 App 写入（encode 第 89 行 `encode(permissionMode.rawValue, ...)` 写的永远是 String）；要被外力破坏才会出现（用户手改 JSON / 第三方工具）
- 但**用户因为试着改 permission_mode 试图"绕过"修改流程而误手敲数字 → 整个工作区列表炸**这种事，在个人工具语境里也确实可能发生（开发者用户群）
- R1 已注意到 `decodeIfPresent(String.self, ...)` 是正确选择（避免直接 decode PermissionMode 时大小写错也炸），但**还没解决"类型不是 String"这一最后路径**

**🟡 建议**（不阻塞）：在 Tab.init(from:) 给 permissionMode 解码外层加 try? 容错：

```swift
// 极端容错：哪怕 permission_mode 字段类型是 number/boolean，也不让整个 Tab 解码失败
let raw = try? c.decodeIfPresent(String.self, forKey: .permissionMode)
if let raw = raw, let m = PermissionMode(rawValue: raw) {
    self.permissionMode = m
} else {
    self.permissionMode = .default
}
```

这样可保证："**任意 permission_mode 字段值都不会让 Tab 解码失败**"，与变更说明 §3.2 "缺失字段 → default"的设计意图一致。

**整体结论**：兼容性绝大多数路径 OK，仅"字段类型错"是个理论缺陷。**通过**，建议小修。

### ✅ 2. 并发安全 — MainActor 是否够

代码中所有 mutation 入口都标注 `@MainActor`：
- `TabManager` class 上 `@MainActor`（TabManager.swift:17）→ `setPermissionMode`、`createTab`、`removeTab`、`persist`、`loadFromDisk` 全部串行
- `ProcessHost` 上也 `@MainActor`（ProcessHost.swift:9）→ `startProcess`、`stopProcess` 串行
- SidebarView / TabStripView 中的 `createTab` / `changePermissionMode` 都从 SwiftUI 主线程触发

**潜在并发场景检查**：

| 场景 | 分析 |
|---|---|
| setPermissionMode 同时 createTab | 都 MainActor，串行执行，不可能交错 ✅ |
| processTerminated 回调过程中调 stopProcess/startProcess | processTerminated 是 `nonisolated`，但内部 `DispatchQueue.main.async + MainActor.assumeIsolated`（ProcessHost.swift:251-262）→ 状态变更走 main → 与 setPermissionMode 串行 ✅ |
| asyncAfter(0.2s) 闭包 vs 同时另一处 createTab | 都派发到 main queue → FIFO 串行 ✅ |
| TabSyncBridge.changes.sink 触发 sendTabChanged 是 Task | Task { await self.sendTabChanged(event) }（TabSyncBridge.swift:39-41）→ 异步发送到 WSClient → ws 是 actor / MainActor，不会冲突 ✅ |

**结论**：MainActor 串行模型够用。**通过**。

### ⚠️ 3. 重启时序竞态 — 200ms 期间用户操作

`changePermissionMode` 关键流程（SidebarView:166-172 / TabStripView:113-117）：
```
1. setPermissionMode（model 改 + persist + broadcast）
2. stopProcess(updated.id)  // SIGTERM，PTY clean-up 异步
3. DispatchQueue.main.asyncAfter(0.2s) { startProcess(updated) }
```

**这 200ms 期间用户能做什么 + 后果分析**：

| 用户动作 | 当前代码后果 |
|---|---|
| **A. 点其他 Tab** | `selectedTabId` 改变，但 startProcess 闭包捕获的是 `updated`（值类型 Tab 快照），仍会启 updated.id 的进程。**OK**：用户切回时会看到新 mode 进程已起。 |
| **B. 关闭这个 Tab**（同一个 tab） | confirmAndClose → `processHost.stopProcess(tab.id)`（无-op，已 stop）→ `removeTab(tab.id)`，从 tabs 数组移除。0.2s 后闭包内 `startProcess(for: updated)`：检查 `terminalsByTab[tab.id] != nil` → false（已被移除），创建新 view，最终 `tabManager?.updateStatus(tab.id, status: .running)`，但 tab 已被 removeTab 删除 → updateStatus（TabManager:133-143）`guard let idx = ...else { return }` 静默退出。**遗留**：terminalsByTab 中有一个 view 引用一个已删除 tab 的进程；该 view 永远不会被 stopProcess（tab.id 不存在了）。⚠️ **潜在资源泄漏 + 僵尸 claude 进程**。 |
| **C. 再次切 mode**（同一 tab） | setPermissionMode 改值（旧 mode 已被 setPermissionMode 改成 modeA，现在改成 modeB；guard 检查 != 通过）→ persist → stopProcess（terminalsByTab 已无该 tab，no-op）→ asyncAfter 第二次 startProcess(updated_with_modeB)。同时第一次的 asyncAfter 仍在排队，0.2s 后执行 startProcess(updated_with_modeA)。**两个闭包顺序触发**：第二个先排好的先到、第一个后排但延迟相同，主队列 FIFO，先入队先出队 → **闭包 1 先跑（startProcess modeA），闭包 2 后跑（startProcess modeB），但闭包 2 内部 `if terminalsByTab[tab.id] != nil { return }` 会判 true（闭包 1 刚创建）→ 直接 return**。**最终该 Tab 跑着 modeA 的 claude，但 model 里持久化的是 modeB**。⚠️ **严重不一致**。 |
| **D. 用户在中间手动从外部杀 claude 进程** | processTerminated 回调更新 status；后续 asyncAfter 仍会 startProcess。**OK**。 |

**关键风险**：场景 C（200ms 内连续切 mode）和场景 B（200ms 内关 tab）。

C 是更可能触发的：用户切错了 mode 想立即改回，UI 完全不阻止两次连续点击（contextMenu 是无状态的）。

**🟡 建议**：
1. 维护一个 `pendingRestart: Set<UUID>` 集合，进入 changePermissionMode 时 insert，0.2s 闭包末尾 remove，期间该 tab 的 changePermissionMode 直接弹"正在切换中…"
2. 或者更简单：startProcess 闭包内不要 `if terminalsByTab[tab.id] != nil { return }` 静默，而是先 stopProcess 再 startProcess。这样场景 C 中闭包 2 会覆盖闭包 1 的 modeA 进程，最终匹配 model 的 modeB

场景 B 的资源泄漏：在闭包内加 `guard tabManager?.tab(by: updated.id) != nil else { return }` 即可。

### ⚠️ 4. claude 子进程不接受新 mode 怎么办

**当前行为**（startProcess ProcessHost.swift:34-89）：
- claude binary 找不到 → 显式 updateStatus(.error, errorReason)，UI 有 banner（R1 已确认）✅
- claude binary 存在但参数错（如 claude 升级后 `--permission-mode` 改名/删除/不支持新 mode 值）→ claude 进程会立即 exit non-zero → SwiftTerm processTerminated 回调 → ProcessHost.swift:249-263 处理：
  - `updateStatus(id, status: .error, exitCode: exitCode)`
  - **但没有 errorReason**（passed nil）→ Tab.swift:39 errorReason 注释说"errorReason 仅在 reason 比 generic 'process exited' 更具体时填"
  - UI 看到的是：Tab 状态变 error，exitCode != 0，**没有"是不是 --permission-mode 不识别"的提示**
  - 用户看到 banner / terminal 里的 stderr 输出（claude 自己应该会打"unknown option --permission-mode"），但日志里只有 `process exited tab=xxx exitCode=2`

**潜在问题**：
- claude CLI 未来某版本删除 `dontAsk` 或重命名 `bypassPermissions` → 用户该 mode 的 tab 全部启动失败，但 UI 不会提示原因
- 用户重启 App，旧 mode 持久化值 `dontAsk` 仍在 → 每次都失败
- 没有"启动失败时自动 fallback 到 default 重试"机制

**严重度评估**：
- 极低频（claude CLI 通常向后兼容 flag 值），但一旦发生影响所有非 default Tab
- 用户能在 SwiftTerm 终端区域看到 claude 的 stderr 输出（PTY 直接 feed 到 view），所以**不是黑盒**，只是没有结构化 UI 提示

**🟡 建议**（不阻塞，但作为防御性设计）：
- 在 ProcessHost 监听 processTerminated 时，如果 exitCode != 0 且**进程存活时间 < 1s**（说明是参数错误，不是用户自己 exit），设置 errorReason = "Claude 子进程启动失败（exit code N）。可能是 --permission-mode 值不被支持，请尝试切回 default 模式。"
- 或者在 changePermissionMode 完成 0.2s 重启后，再过 2s 检查 status 是否变 error，如果是则弹一次性 NSAlert 提示"切换失败"

**当前实现"够用但不友好"**。**通过**（个人工具语境，开发者用户能看终端 stderr）。

### ⚠️ 5. 业务影响 — 用户预期与默认行为

#### 5.1 既有 Tab 升级到新版本 → 默认 default

**变更说明 §3.2 "原有 Tab 升级后自动按 default 模式启动"** 与**老版本的实际行为**对比：

- 老版本：claude 启动**不带** `--permission-mode` flag → 走 claude CLI 自己的默认（实测 claude --help 显示 default 是 CLI default）
- 新版本：升级后老 Tab 重启时显式传 `--permission-mode default`

**结论**：行为**实际未变化**（claude CLI 默认就是 default）✅。"用户原以为是无门槛"的假设**不成立** —— 老版本本来就是 default，用户既然能用，那一定接受 default 的逐项确认体验。

**但**：banner 文字（ProcessHost.swift:74）现在多了 `--permission-mode default` —— 用户首次看到这个 banner 可能会问"我什么时候选过 default？"。

**🟡 建议**：在 banner 后附一句 `(可右键工作区切换)`，让升级用户知道 mode 是可控的。或者在 changelog / 启动 once-only 弹窗里提示。

**严重度低**，**通过**。

#### 5.2 bypassPermissions 高危选项无二次确认

`bypassPermissions` 文档明确"仅适合隔离容器 / VM"（PermissionMode.swift:38），但当前 UI 上：

- **创建工作区时**选择 bypassPermissions → 直接确定 → 创建并启动，无额外确认
- **修改时**选 bypassPermissions → 弹一次"切换并重启"通用确认 → 用户点确认 → 立即生效

**对比 macOS 系统其他高危操作**（如 `sudo`、Full Disk Access）都需要二次输密码 / 二次确认。

**变更说明 §5 明确写「不做"mode 切换的二次确认弹窗序列（一次确认即够）"」** —— 但 §5 的"二次确认"指的是 mode 切换整体；并未排除"对 bypassPermissions 这一特殊 mode 单独加警告"。

**潜在业务影响**：
- 用户在非隔离环境（直接本机）选 bypassPermissions → Claude 拥有所有文件系统读写 + 命令执行权 → 误操作（rm -rf）或被 prompt injection 时无防线
- 个人工具语境下，用户=开发者，风险自担，但**至少日志要打 warning 让事后审计有迹可循**

**🟡 建议**：
1. **必做**：`setPermissionMode` 中如果 mode == .bypassPermissions，`log.warn(...)` 替代 `log.info(...)`（TabManager.swift:83）。同 `createTab` 第 63 行
2. **建议做**：选 bypassPermissions 时弹的 NSAlert 顶部加红字「⚠️ 此模式将允许 Claude 执行任意命令，**仅在隔离环境**使用」+ 默认按钮设为"取消"而非"切换"
3. **可选**：完全允许选 bypassPermissions，但单独弹"我知道风险"复选框 NSAlert

不阻塞，**建议至少做日志 warn**。

#### 5.3 .renamed 事件触发 ws 广播 → Android UI 重绘

承接 R1 已发现：`.renamed` 触发 TabSyncBridge.sendTabChanged → broadcast tab.changed action=renamed → 手机端收到。

**TabSummary 字段**（ProtocolMessage.swift:67-91）：
```
id, name, folder, claudeStatus, lastActivityAt
```
**不含** permission_mode。换 mode 后 TabSummary 各字段（除了 status 在 stop/start 间会变 idle/running，但这是 status 推送独立通道）**没变化**。

**Android 端收到一条 action=renamed 的 tab.changed**：
- name 没变 → diff 后无新内容 → 列表 row 不会重渲染（如果 Android 用 key/equals 比较）
- 但 ws 报文确实发了一条，**Android 客户端的协议层会 parse → 派发到 controller → state 比较 → 不变 → no-op**

**影响评估**：
- 一次 ws 推送 ≈ 几百字节，开销可忽略
- Android 端有 build_widget 函数 diff 机制 → 视 listener 实现，若是 RiverpodNotifier 用 `state = newValue` 而 newValue ≠ 旧 state → 列表会触发 rebuild（但内容相同 → 用户看不见变化）
- 一次 mode 切换：stopProcess（status idle 推一次）+ setPermissionMode（.renamed 推一次）+ startProcess（status running 推一次）= 3 次 ws 推送。可接受

**结论**：**功能无害**，只是冗余推送一条 no-op tab.changed。R1 已记为信息项，本轮维持判断。**通过**。

### ✅ 6. 资源 — askPermissionMode 调用频率与泄漏

调用点：
- 创建工作区：每次新建 1 次（SidebarView/TabStripView createTab）→ 低频
- 修改 mode：不调用 askPermissionMode（直接通过 contextMenu 选项）→ 0 次

**askPermissionMode 每次创建的对象**：
- `NSAlert`（栈临时）
- `NSPopUpButton`、`NSTextField summary`、`NSView container`（栈临时）
- `PermissionModePopupTarget`（class instance，被 `withExtendedLifetime` 强引用至 runModal 结束 → modal 结束 ARC 释放）

**潜在泄漏点**：
- `popup.target = target`（AppKit target/action 系统的 target 是 **weak**） → 即便 popup 还存在但 target 已释放也安全 ✅
- container.addSubview 引用 popup + summary → container 释放时 view tree 释放 ✅
- alert.accessoryView = container → alert 释放时 accessoryView 释放 ✅

`runModal()` 同步阻塞 → 返回时整个对象图都在栈作用域，ARC 立刻释放。**无泄漏**。

**多次连续创建**（用户反复点 + 然后取消，反复创建工作区）：
- 每次 askPermissionMode 调用结束都释放所有对象
- AppKit 没有 NSAlert 缓存池
- 无泄漏 ✅

**通过**。

### ⚠️ 7. UI 兼容性 — SwiftUI Menu + Label/Text + ForEach 在不同 macOS 版本

代码（SidebarView.swift:50-61 / TabStripView.swift:27-37）：
```swift
Menu("权限模式") {
    ForEach(PermissionMode.allCases, id: \.self) { m in
        Button(action: { ... }) {
            if m == tab.permissionMode {
                Label(m.displayName, systemImage: "checkmark")
            } else {
                Text(m.displayName)
            }
        }
    }
}
```

**macOS 14 (Sonoma) / 15 (Sequoia) / 26 (假设) SwiftUI 行为差异分析**：

| 元素 | macOS 14 | macOS 15+ |
|---|---|---|
| `Menu` in contextMenu | 显示为子菜单 ✅ | 同 ✅ |
| `Label(text, systemImage:)` in Button in Menu in contextMenu | **历史 issue**：有时只显示 text，systemImage 被吞 | macOS 15 起 SwiftUI 修了部分 issue，但 contextMenu 子菜单仍是已知重灾区 |
| `Text` in Button | 始终正常 ✅ | 同 ✅ |
| ForEach + CaseIterable | 正常 ✅ | 同 ✅ |

**潜在问题**：
- 用户在 macOS 14 上看到的可能是 6 个 plain text 选项，**完全分不清哪个是当前 mode**（checkmark 没渲染）
- 这是 R1 第 9 项已发现的问题，R1 标记 🟡 建议
- 本轮放大风险评估：**当 checkmark 不显示，用户唯一区分手段就是"切换并重启"对话框里的"从 X 改为 Y"提示**。点选项前确实无法知道当前选中的，**实际功能可用性下降但不破坏**

**🟡 建议**（与 R1 一致，本轮加权）：
改用纯文本前缀方案：
```swift
Button(action: { ... }) {
    Text((m == tab.permissionMode ? "✓ " : "    ") + m.displayName)
}
```
跨版本稳定，无对齐问题。

**严重度**：中（影响用户判断当前 mode 的能力，但有其他途径补救）。**通过**（不阻塞），但**强烈建议 R3 实测验证 + 修复**。

### ⚠️ 8. 极端输入 — Modal 中断路径

`alert.runModal()` 在 NSAlert 上的返回值类型 `NSApplication.ModalResponse`：

| 用户动作 | runModal 返回值 |
|---|---|
| 点"确定" | `.alertFirstButtonReturn`（1000） |
| 点"取消" | `.alertSecondButtonReturn`（1001） |
| 按 ESC 键 | 同"取消"（NSAlert 内置 ESC 绑定到 cancel button） → `.alertSecondButtonReturn` |

**代码处理**（TabUIHelpers.swift:55）：
```swift
guard response == .alertFirstButtonReturn else { return nil }
```

- 任何非 .alertFirstButtonReturn 都返回 nil → 上层取消 ✅
- ESC / 点取消 → 都返回 nil ✅
- 不存在"alert window 关闭按钮" → 不需处理

**changePermissionMode 中的 NSAlert**（SidebarView.swift:159-165）：
```swift
guard alert.runModal() == .alertFirstButtonReturn else { return }
```
同样处理 ✅

**结论**：所有中断路径覆盖完整。**通过**。
