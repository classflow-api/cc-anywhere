### 🟡-1 HistoryBridge.allHistoryFiles 不区分 agent-*.jsonl 归属，跨会话内容污染历史回放 — 已修复

**定位**：`MacClient/Sources/CCAnywhere/Services/HistoryBridge.swift:217-254`

**修复内容（处置 A）**：

在 `allHistoryFiles` 拿到 `latestParent` 后，对所有 `agent-*.jsonl` 做 mtime cutoff 过滤：仅保留 mtime ≥ 父 session mtime - 24h 的子 agent 文件。父 session mtime 不可得时退化为全集（保持与改造前行为一致）。

```swift
let cutoff: Date? = latestParent.flatMap {
    try? $0.resourceValues(forKeys: [.contentModificationDateKey]).contentModificationDate
}.flatMap { $0.addingTimeInterval(-24 * 3600) }
let agents = urls.filter { url in
    let name = url.lastPathComponent
    guard name.hasSuffix(".jsonl"), name.hasPrefix("agent-") else { return false }
    guard let cutoff = cutoff,
          let mtime = (try? url.resourceValues(forKeys: [.contentModificationDateKey]))?.contentModificationDate else {
        return true  // 父 session mtime 取不到 → 不过滤，与旧行为一致
    }
    return mtime >= cutoff
}
```

**复审独立验证**：

| 验证维度 | 结论 |
|----------|------|
| 父 mtime 不可得时退化为全集 | ✅ guard 双层 nil 路径返回 true，保持旧行为；不破坏 R-F8-001 |
| 24h 时间窗合理性 | ✅ 子 agent 通常在父 session 创建后几分钟到几小时内活跃；24h 涵盖跨日对话（如夜间运行的 Task）；6+ 月累积场景被裁掉 |
| mtime 比较语义 | ✅ agent-*.jsonl mtime 是子 agent 最后写入时间，必 ≥ 父 Task tool_use 创建时间（即 ≥ 父 session 当时 mtime - 任何正数 → 必 ≥ 父 mtime - 24h） |
| 残留污染风险 | 🟢 当日内多会话理论上仍纳入，但 promptHash 跨会话冲突概率近 0（自然语言 prompt）+ phone 端 uuid dedup + tail-limit cap 兜底；不构成新 🟡 |

### 🟡-2 父 session 的 Task tool_result 在主流与折叠块内双重渲染 — 已修复

**定位**：`AndroidClient/lib/data/chat_repository.dart:606-661`

**修复内容（处置 A）**：

在 `_preprocessSubAgent` 内引入 `absorbedByBlock` 标志位。检测到 `toolResult` 且 `blocks.containsKey(refId)` 命中时设置 `absorbedByBlock = true`，跳过 `passthrough.add(raw)`。原 `passthrough.add(raw)` 移到 `_detectTaskInRecord` 之后、按 `absorbedByBlock` 决定。

```dart
var absorbedByBlock = false;
final detection = _detectTaskInRecord(raw);
if (detection != null) {
  if (detection.kind == _TaskRecordKind.toolUse) {
    // ... 建/回填折叠块 ... (不改 absorbedByBlock，主流仍渲染 ToolUseCard)
  } else if (detection.kind == _TaskRecordKind.toolResult) {
    final refId = detection.toolUseId;
    if (refId != null && blocks.containsKey(refId)) {
      final block = blocks[refId]!;
      block.finalResult = raw;
      block.status = detection.isError == true ? 'failed' : 'done';
      absorbedByBlock = true;  // R-F4-002 隐含语义：final 是折叠块组成部分
    }
  }
}
if (!absorbedByBlock) {
  passthrough.add(raw);
}
```

**复审独立验证**：

| 验证维度 | 结论 |
|----------|------|
| 探测后 passthrough.add 顺序 | ✅ detection 后才决定，逻辑正确 |
| 仅命中场景 absorb（toolResult + blocks.containsKey） | ✅ 严格 AND 条件；refId 为 null 或未知 block 时仍 passthrough |
| 普通 message（detection==null）影响 | ✅ 跳过 if 块，absorbedByBlock 保持 false，照常 passthrough |
| 普通工具的 tool_result（非 Task）影响 | ✅ refId 不在 blocks → 不 absorb；与 ToolResultCard 主流渲染保持一致 |
| Task tool_use 自身是否引入新双重渲染 | 🟢 见 🟢-5 — 主流 ToolUseCard + 折叠块 placeholder 两卡共存，但视觉信息互补（一为"工具调用动作"，一为"进度追踪器"），主观偏好级，未升级为 🟡 |

## 🟢 绿色（保留）

### 🟢-1 promptHash 短 prompt 误匹配理论可能（继承初审）

200 字符 SHA1 在自然语言 prompt 下冲突概率近 0；7 字符 hex agentId（2.7 亿空间）在单会话活跃集合内基本无冲突。**主观顾虑，非阻塞**。

### 🟢-2 子 agent 第一个 tool_approval 卡片可能丢失 chip（继承初审）

子 agent 启动 → 第一个 PreToolUse hook 可能比 agent-*.jsonl 首条落盘更早（理论上）→ `findSubAgent` miss → tool_approval 卡片不带 chip（但仍正常显示 "Bash xxx 批准吗"）。

不会触发**错误标记**（isFromSubAgent 仅在反查成功才置 true），只是退化为"应有的 chip 漏标"。实测 JSONL 首条延迟 29ms，绝大多数情况下子 agent thinking + 第一次 tool_use 远晚于 29ms。**已知工程限制**。

### 🟢-3 silentInitialScan 与 FSEvent 启动后微秒级 race（继承初审）

`watch()` 第 471 `FSEventStreamStart` 后理论窗口内 FSEvent callback 可能先入 queue 后 silentInitialScan 入队。由于 queue 是 serial，二者串行化执行；且二次扫描通过 `existing?.parentToolUseId == nil` 守门、`seenUuids` 去重 → 完全幂等。无功能影响。

### 🟢-4 HistoryBridge.promptHash 注释误导（继承初审）

`HistoryBridge.swift:204-211` 注释自承"算法与 JSONLWatcher.promptHash（SHA1）独立，无需一致"，实际是 Swift Hasher 不是 SHA1。功能上 HistoryBridge 内部 join 自洽（不跨进程比较 hash），仅算法名称表述。第一轮已记录，本轮不强制处置。

### 🟢-5 Task tool_use 在主流 ToolUseCard 与折叠块 placeholder 并存（本轮新增）

**定位**：`chat_repository.dart:613-645`

🟡-2 修复仅对 tool_result 做 absorb；Task tool_use raw 仍走 passthrough → 主流照常渲染 ToolUseCard（"Claude 调用了 Task 工具"），同时折叠块 placeholder 渲染 SubAgentFoldedBlock（"⚡ Task 子 agent + summary + 步数 + 状态"）。

**判定为 🟢 而非 🟡 的理由**：

1. PRD § 4 流程图说"创建 SubAgentFoldedBlock(T1)"，**未明示**禁止主流保留 Task tool_use 卡片
2. 两卡片信息维度互补：ToolUseCard = 时间点动作；SubAgentFoldedBlock = 持续进度追踪
3. 初审 22 项清单已隐含接受该设计（仅 tool_result 双重渲染列为 🟡-2，未列 tool_use）
4. 与 5000 字 tool_result 重复（明显 UX 问题）不同，Task tool_use 的 input.prompt 与折叠块 promptSummary（60 字截断）虽有重叠，但前者展示完整工具调用 metadata、后者展示状态卡，**视觉冗余但不致命**
5. 修复需求范围已超出本轮 🟡-2 处置（需触及 ToolUseCard 路由 / 主流过滤逻辑，改动面 > 50 行），不属于 review-2 处置范围

**主观偏好**：未来产品验收若提出可单独走 L1/L2 流程优化（折叠块取代主流 Task tool_use 卡）。
