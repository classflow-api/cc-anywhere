## 关键发现 / 待办

### ⚠️ Step 5（需批准 UI）— 测试环境不适用
当前 Claude Code 会话的权限模式不会拦截 Bash 命令（包括 `curl ... | bash` 和 `sudo rm` 都直接执行）。**未能触发**「待批准」UI 测试。

**建议**：在 cc-anywhere 端启动一个 `default` 或 `plan` 权限模式的会话再重测此项。

### ⚠️ Step 8（Glob/Grep）— 工具未加载
当前会话工具列表中没有 Glob 和 Grep（不在已加载工具列表，也不在 deferred tools 列表）。用 Bash 的 `find`/`grep` 模拟，但 tool_use 协议名变成了 Bash。

**建议**：cc-anywhere 端检查 Claude Code 实例的工具集配置，确保 Glob/Grep 启用，才能完整测试 tool_use 协议覆盖。

### ✓ Step 14（代码块语法高亮）— 复测通过
**现象**：用 Python / Dart / TypeScript / Bash / SQL 五种语言代码块复测，手机端全部正常显示：
- 每个代码块独立卡片，不混入正文
- 等宽字体（monospace）
- 暗色背景，与正文气泡区分
- 关键字 / 字符串 / 注释 / 函数名均有颜色高亮
- 右上角有复制按钮

**结论**：本项无需改进。

（备注：第一次测试用了过短的代码块（如 `def hello(): print("Hello")` 单行），可能因关键字密度低导致未察觉高亮。复测用 5 种语言的真实代码片段后确认高亮完整。）

---

### ⚠️ Step 12（长文本流式）— 流式渲染未实现
**现象**：手机端收到约 1000 字的长文本回复时，是**一次性整段显示**，而不是 Mac 端那样的「逐 chunk 流式更新」。

**影响**：
- 长回复用户感知不到「Claude 在思考/输出」的进展，体验类似 HTTP 请求的整体响应
- 在弱网或慢响应场景下，用户可能误以为系统卡死
- 与验证需求 24（性能优化）的目标不符

**建议**：
- cc-anywhere Host 端需把 Claude Code 的 `assistant.delta` / `content_block_delta` 事件按 chunk 推到手机
- 手机端 UI 需要支持「同一个消息气泡的内容增量追加 + 自动滚到底」
- 评估 WebSocket 消息聚合策略，避免每个 token 都发一次造成网络压力（典型做法：按时间窗或字数窗 flush）

### ✗ Step 11（AskUserQuestion）— ⭐ 核心缺陷
**现象**（两次测试共同确认）：
- 第一次测试（4 个选项问题）：Mac 端调用 `AskUserQuestion` 后，**手机端完全没出现**问题文本和选项按钮，必须在 Mac 端选择。Mac 端选完后，手机端**才**补出一张「工具结果」卡片
- 第二次复测（4 个选项问题）：用户在手机端选择「**什么都没看到**」，再次确认手机端**根本未展示任何 AskUserQuestion 相关 UI**（没有问题文本、没有选项按钮、没有任何卡片）

**根因推测**：
- cc-anywhere 协议层没有把 `tool_use(name=AskUserQuestion)` 事件转换成专门的「问题消息」推送
- 或者推送了但手机端 UI 没识别这个工具，被丢弃
- 只有 `tool_result`（回答后的结果）能流到手机端，但已失去交互意义

**影响**：手机端无法参与 `AskUserQuestion` 交互，所有需要用户决策的场景都退化到 Mac 端。这是 cc-anywhere 移动端的功能缺口。

**建议**：
- 在 cc-anywhere 协议层新增对 `AskUserQuestion` 工具调用的特殊处理
- 监听 `tool_use(name=AskUserQuestion)` 事件，把问题 + 选项流到手机端
- 手机端渲染交互卡片（问题文本 + 多个选项按钮 + Other 输入框）
- 用户点击选项后，把结果作为 `tool_result` 推回 Mac 端 Claude
