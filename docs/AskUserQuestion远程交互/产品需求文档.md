## 2. 背景与目标

### 2.1 业务背景

cc-anywhere 当前的"JSONL 旁观者"架构存在三个核心限制：

1. **AskUserQuestion 决策只能在 Mac 端做**：手机端只能在 Claude SDK 把 `tool_use + tool_result` 落盘到 JSONL **之后**看到事后记录，无法参与决策。
2. **长文本 / 工具调用进度无实时反馈**：手机端要傻等到 JSONL 落盘才能看到 Claude 在做什么。
3. **危险工具（Bash/Write）批准无法远程化**：Mac TUI 弹窗，手机端看不到。

零散需求批次 #1-#37 完成后的验收测试明确暴露上述问题。

### 2.2 项目目标

让手机端真正成为"远程决策端"，而不只是"事后查看器"：

| 能力 | 改善前 | 改善后 |
|---|---|---|
| AskUserQuestion 决策 | 事后查看 | 实时弹卡片决策 |
| 工具调用进度 | 落盘后才知道 | "正在执行 Read xxx.md" 实时显示 |
| 待批准的 Bash/Write | Mac 端弹窗 | 远程批准/拒绝 |
| Claude 状态变化 | 推断 | 实时通知 |
| Phone Card 自定义输入 | 缺失 | 支持 Other 文字输入框 |

### 2.3 成功指标

- **G1 (P0)**: Mac 端 Claude 触发 AskUserQuestion 后，phone 端在 **1 秒内**弹出问题卡片（端到端延迟）。
- **G2 (P0)**: phone 端选答后，Claude SDK 在 **2 秒内** 拿到 answers 并继续。
- **G3 (P0)**: 用户在终端直接跑 `claude`（不经过 Mac App）时，**不被本需求的 hook 任何干扰**（hook 软失败，等价于"没装 hook"）。
- **G4 (P0)**: Mac App 关闭"启用远程 hook"偏好开关后，`~/.claude/settings.json` 中 cc-anywhere 的 hook 条目被完整卸载。
- **G5 (P1)**: phone 与 Mac App 同时显示同一张 AskUserQuestion 卡片，任一端 wins，另一端在 500ms 内显示"已被 X 回答"。
- **G6 (P1)**: phone 全部离线时，hook 在 Mac App inner timeout（默认 5 分钟）后返回 deny，Claude TUI 走 fallback 弹原问题，不卡死。

## 4. 功能清单

### F1 (P0) — Hook 基础设施 + AskUserQuestion 实时交互（对应 M1）

- **F1.1**: hook bridge 脚本（Python，~200 行）包含 `ask` / `progress pre` / `progress post` / `notification` 四子命令；前置环境变量 + 软失败保护三道门。
- **F1.2**: Mac App `HookIpcServer`（Unix socket server，路径 `~/Library/Application Support/cc-anywhere/hook.sock`），含 tab 路由（by `CC_ANYWHERE_TAB_ID` env）、请求登记表、超时回收、winner 锁。
- **F1.3**: Mac App `SettingsJsonInstaller` —— 启动时弹许可弹窗，同意后用 jq 模式安全 append `~/.claude/settings.json`；偏好开关关闭时只删 cc-anywhere 自己那条。
- **F1.4**: Mac App 子进程启动时注入 `CC_ANYWHERE_TAB_ID=<tab_uuid>` 环境变量。
- **F1.5**: 协议 type `ask.question.pending` / `ask.question.answer` / `ask.question.timeout`（详见技术实施文档第 6 节）。
- **F1.6**: Phone 端 `AskUserQuestionCard` 改造为实时模式：接收 `ask.question.pending` → 立即弹卡片；选项点击 → 发 `ask.question.answer`；接收 `ask.question.timeout` → 撤销卡片。
- **F1.7**: Mac App 内 `AskQuestionCard` SwiftUI 视图（非 Claude TUI 弹窗）：phone + Mac 同时看到同一张卡片，winner 锁仲裁。
- **F1.8**: Phone 端 / Mac App 端 AskQuestionCard 均支持 **"自定义文字回答"输入框**（对齐 AskUserQuestion 工具自带的 Other 选项行为）。
- **F1.9**: phone 收到 `ask.question.pending` 但已被回答 → 卡片显示"已被 X 回答（label/自定义文字）"，不允许重复提交。

### F2 (P0) — 工具进度推送（对应 M2）

- **F2.1**: hook bridge `progress pre` / `progress post` 子命令异步推送，立即返回 `{}`（不阻塞 Claude）。
- **F2.2**: 协议 type `tool.progress.pre` / `tool.progress.post`。
- **F2.3**: Phone 端 `ToolProgressIndicator` —— 消息列表底部显示"⚡ 正在执行 [tool_name] [tool_input 摘要]"，PostToolUse 收到后消失（success：自然消失 / error：变红色 toast 提示）。
- **F2.4**: `tool.progress.*` 必须包含 `tool_use_id`，phone 端通过 `tool_use_id` 跟 JSONLWatcher 后续事件去重。

### F3 (P1) — Notification 推送（对应 M3）

- **F3.1**: hook bridge `notification` 子命令。
- **F3.2**: 协议 type `notification`。
- **F3.3**: Phone 端 `NotificationToast` —— 顶部 snackbar 显示 Claude 状态变化（如 idle / permission_prompt）。

### F4 (P1) — 危险工具远程批准（对应 M4）

- **F4.1**: PreToolUse hook matcher 扩展到 `Bash|Write|Edit`，走 `ask` 通道（复用 F1.2 socket 通信）。
- **F4.2**: AskQuestionCard 渲染时检测 `tool_name`，若属于 Bash/Write/Edit → 顶部加红色徽章"⚠ 工具批准"+ 显示 `tool_input` 摘要（如 `bash -c "rm -rf /tmp/xxx"` 截断 200 字符）。
- **F4.3**: 用户选"允许"→ hook 返回 `{permissionDecision: "allow"}`；选"拒绝"→ 返回 `{permissionDecision: "deny"}`。
- **F4.4**: M4 默认关闭，由偏好开关"启用工具批准远程化"独立控制（M1-M3 跟它解耦）。

### F5 (P0) — 双推去重

- **F5.1**: JSONLWatcher 解析 JSONL 时跳过 hook 已推送过的事件类型（`AskUserQuestion` / `Bash` / `Write` / `Edit` 这些被 hook 推送的工具调用，按 `tool_use_id` 维护去重表）。
- **F5.2**: Phone 端使用 `tool_use_id` 作为去重 key 二次保护，防止补拉场景下的双显示。

### F6 (P0) — 用户控制

- **F6.1**: Mac App 偏好面板加 2 个开关：
  - "启用远程 hook（M1-M3）"，**默认关闭**（首次启动不主动修改 `~/.claude/settings.json`，遵循 NFR-U1/U2 的零侵入原则；用户主动 OFF→ON → 弹许可弹窗 → 同意后 install）
  - "启用工具批准远程化（M4）"，默认关闭，依赖主开关（R-F6-003）
- **F6.2**: 主开关关闭 → `SettingsJsonInstaller` 立即卸载 cc-anywhere 的 PreToolUse / PostToolUse / Notification 条目。
- **F6.3**: 主开关首次 OFF→ON → 弹许可弹窗，用户同意后 append；之后 OFF→ON 不再弹（许可标志 `didShowHookInstallAlert` 持久化，一次性同意）。

## 5. 非功能性需求

### 5.1 性能

- **NFR-P1**: hook bridge 脚本启动开销 ≤ 100ms（Python `python3` 启动 50ms + 脚本初始化 50ms 预算）。
- **NFR-P2**: Mac App socket → ws 推送链路单跳 ≤ 50ms（局域网/同 host 推 server，server 推 phone）。
- **NFR-P3**: 端到端 AskUserQuestion 触发 → phone 弹卡片延迟 ≤ 1 秒（含 hook 启动 + IPC + ws + phone 渲染）。
- **NFR-P4**: Phone 答题 → Claude SDK 拿到 answers 延迟 ≤ 2 秒。

### 5.2 安全

- **NFR-S1**: socket 文件权限 0600（仅 owner 可读写）。
- **NFR-S2**: `CC_ANYWHERE_TAB_ID` 环境变量值为 UUIDv4 随机字符串，不可枚举。
- **NFR-S3**: hook bridge 脚本不向 phone 推送任何 env / cwd 之外的进程信息（防信息泄漏）。
- **NFR-S4**: `~/.claude/settings.json` 写入前必须 backup 到 `~/Library/Application Support/cc-anywhere/settings.json.bak.<timestamp>`，最多保留最近 5 份。
- **NFR-S5**: settings.json 写入操作必须先 jq 解析验证 JSON 合法 → 写入临时文件 → 原子 rename，禁止直接覆盖原文件。
- **NFR-S6**: settings.json 卸载操作必须**精准识别** cc-anywhere 注入的条目（通过 command 路径前缀 `cc-anywhere-hook-bridge` 匹配），不得误删其他 plugin 的 hooks。

### 5.3 兼容性

- **NFR-C1**: macOS 13+（与 cc-anywhere Mac 端最低支持版本一致）。
- **NFR-C2**: Claude CLI 任意版本（hook 协议为 Claude 官方稳定接口）。
- **NFR-C3**: Phone 协议向前兼容 —— 旧版 phone 收到未知 type 静默丢弃，不崩溃。
- **NFR-C4**: 与现有 yoolines-dev-workflow / superpowers / episodic-memory 等 plugin 的 settings.json hooks **共存**（append 模式，不覆盖现有数组）。

### 5.4 可用性

- **NFR-U1**: hook bridge 脚本异常（崩溃 / 超时 / socket 不可达）**永远不能比"没装 hook"更糟**，所有失败路径必须输出 `{}` 让 SDK 走 fallback。
- **NFR-U2**: 用户在终端直接跑 `claude`（无 `CC_ANYWHERE_TAB_ID`）时，hook bridge 必须立刻输出 `{}` 退出，0 干扰。
- **NFR-U3**: hook bridge 错误诊断日志写入 `~/Library/Logs/cc-anywhere/hook-bridge.log`（stderr 落盘），便于 troubleshoot。
- **NFR-U4**: Mac App 偏好开关"启用远程 hook" 关闭 → 立即生效，无需重启。

### 5.5 可观测性

- **NFR-O1**: Mac App `HookIpcServer` 必须在 cc-anywhere 主日志中打印：socket 创建、连接建立/关闭、每条 hook 请求的 tab_id / tool_name / 处理结果 / 耗时。
- **NFR-O2**: Phone 端必须在 console / 日志中打印收到的 `ask.question.pending` / `tool.progress.*` / `notification` 事件（开启 debug 模式时）。

## 7. 风险与依赖

### 7.1 已知风险

| 编号 | 风险 | 概率 | 影响 | 对策 |
|---|---|---|---|---|
| R1 | Claude SDK 不采纳 hook 的 `updatedInput.answers`，TUI 仍弹原问题 | 中 | 高（M1 失效） | 阶段七开发第一件事二次验证；不通过则升级方案 |
| R2 | settings.json 写入与其他 plugin 的 hooks 冲突 | 低 | 中（破坏用户其他 plugin） | jq 安全 append + backup + 精准识别卸载（NFR-S4/S5/S6） |
| R3 | hook 全局生效，误伤用户终端直接跑 claude 的体验 | 低 | 高（破坏日常用法） | hook bridge 三道软失败保护（NFR-U1/U2） |
| R4 | Mac App 单点（崩溃 → socket 不可达 → 所有 Claude session 卡住） | 低 | 高 | hook bridge 软失败放行；Mac App 崩溃监控（后续需求） |
| R5 | phone 离线 → hook 长时间阻塞 → Claude 卡住 | 中 | 中 | Mac App inner timeout（默认 5 分钟，可配） + Claude SDK 走 fallback |
| R6 | Phone Other 输入超长导致 protocol payload 过大 | 低 | 低 | answers 字段 server 限长 8KB，超过返回 400 |

### 7.2 外部依赖

- **D1**: Claude CLI 的 settings.json PreToolUse hook 机制（已通过 echo 实验 2026-05-15 验证打通）。
- **D2**: macOS `/usr/bin/python3` 系统预装。
- **D3**: `~/.claude/settings.json` 写权限（macOS 用户 home 目录默认可写）。

### 7.3 降级方案

- **DG-1**: 阶段七二次验证失败（R1 命中）→ M1 改用 SDK 路线（即可行性评估的方案 A，3-5 人日工作量），但失去 Mac TUI 体验。该决策需在阶段七第一件事确定后由用户拍板。
- **DG-2**: 上线后发现 hook bridge 误伤 → 提供一键卸载脚本 `cc-anywhere-uninstall-hooks.sh`，运行后 settings.json 恢复原状。
- **DG-3**: 协议向前兼容失败（R3 命中） → server 加 protocol 版本协商，旧 phone 走旧路径，新 phone 走新路径。

### 7.4 不在本需求范围

- **OOS-1**: 真 token 级 streaming（逐字流式 assistant 文本）—— hook 不支持 content_block_delta，必须重构成 SDK 路线，失去 TUI。本需求**不包含**。
- **OOS-2**: 跨 Mac / iOS 客户端的统一 hook 中继（本需求只针对 cc-anywhere Mac 客户端）。
- **OOS-3**: hook bridge 多平台分发（Windows / Linux Claude CLI）—— cc-anywhere 当前只在 macOS 上跑。
