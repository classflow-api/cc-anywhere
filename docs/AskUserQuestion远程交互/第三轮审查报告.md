## 性能预算实测预估

对照 NFR-P1/P2/P3/P4 逐项核查代码路径：

### NFR-P1：hook bridge 启动 ≤ 100ms

实际代码路径：
1. python3 解释器启动（macOS 系统预装，~30-50ms 冷启动）
2. `import json/os/socket/sys/traceback/contextlib` — 都是 stdlib，**无重 lib** ✅
3. `must_have_tab_id` 读 env — < 1ms
4. `socket_call` connect → sendall → recv（约 5-15ms 本机 socket，已含 unix domain socket fast path）

**预估总耗时 50-80ms，预算内 ✅**。`safe_exit_with_empty` ctx manager 无明显开销。

### NFR-P2：Mac App socket → ws 推送 ≤ 50ms

实际代码路径：
1. NWConnection.receive 读一行 JSON — < 1ms
2. JSONDecoder.decode HookIpcRequest — < 1ms
3. tabRouter.isActive（NSLock 同步读 Set<UUID>）— < 0.1ms
4. handleProgressPre：`truncateToolInput` 递归对结构截断（深度 1-3 层，字段数 < 10）— < 1ms
5. `await MainActor.run { ws.sendXxx(payload) }` — 主线程切换 + ProtocolMessage 编码 → URLSessionWebSocketTask.send — 通常 < 10ms（localhost ws）

**预估 5-15ms，远低于 50ms 预算 ✅**。

### NFR-P3：端到端 AskUserQuestion → phone 弹卡片 ≤ 1s

链路：hook bridge 启动（80ms）+ socket I/O（10ms）+ Mac actor 处理（5ms）+ ws → server → phone（局域网 30-50ms × 2）+ Phone 渲染（30ms）≈ **200-250ms** ✅

### NFR-P4：phone 答题 → Claude SDK 收 answers ≤ 2s

链路：phone _submit → ws send（30ms）+ server 路由（5ms）+ ws → Mac（30ms）+ resolveAsk + 回写 socket（5ms）+ Python 输出 stdout（< 1ms）+ Claude SDK 处理 hook output（不可控，但通常 < 100ms）≈ **170-250ms** ✅

### Actor 串行化是否瓶颈

`HookIpcServer` actor pendingRequests 操作（注册、resolve、reap、tabRouter）都在 actor 内同步串行。最坏情况：多 phone 竞答 N=10，actor 内 10 次 resolveAsk 顺序执行，每次涉及 `req.continuation.resume` + `await cardController.dismiss` + `await MainActor.run { ws.send... }`。**主要瓶颈在 MainActor 切换**而非 actor 本身；MainActor 上 ws.send 是 async，不阻塞 actor。**实际无明显瓶颈**。✅

### JSONLWatcher.extractToolUseIds 在 burst 场景

`extractToolUseIds(from raw:)` 每行 JSON 解析一次 + 遍历 message.content。在 Claude 单次会话 burst 10 行/秒的场景下，每行 < 1ms（轻量 JSON parse），DispatchQueue 串行不会卡 UI。✅

### Phone DedupService.checkAndMark 性能

- `_evictExpired` 每次调用都遍历 _ttl（24h 内 1k 量级，遍历 < 1ms）
- `unawaited(_persist())` fire-and-forget jsonEncode + setString — 不阻塞业务，但**确实每次 markSync 都会触发一次 SharedPreferences 写**
- 在 hook 通道 burst 场景（连续多个 tool.progress.pre/post），可能引起 SharedPreferences I/O 风暴（Android 上 SharedPreferences commit/apply 是异步的，但仍有磁盘 I/O 排队）

🟢 建议：在 markSync 内做 throttle（如累计 5 个或 1s 触发一次 persist），但当前实测影响有限，可不阻塞上线。

### 性能整体评估

各项 NFR 预算实际有 3-5 倍冗余，**性能层面无阻塞**。
