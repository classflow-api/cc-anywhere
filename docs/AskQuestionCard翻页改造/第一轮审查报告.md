## 修复独立验证结论

### 🔴 问题 1（Enter 提交快捷键）— 已修复 ✓

代码位置：`AskQuestionCardView.swift:300-309`

```swift
.onKeyPress(.return) {
    // 末题且已答完所有问题时 Enter 触发提交；其他情况让事件冒泡
    // （含 TextField 获焦时按 Enter — 此时焦点不在卡片父层，
    // onKeyPress 不会被调用）。
    if isLast && canSubmit {
        submit()
        return .handled
    }
    return .ignored
}
```

独立验证：

- 已在 `body` 父级 VStack 挂 `.onKeyPress(.return)`，与 `.leftArrow` / `.rightArrow` 同一层，焦点协作策略一致
- 触发条件严格为 `isLast && canSubmit`，非末题或未答完时返回 `.ignored` 让事件冒泡，**未在任意题任意状态触发提交**
- TextField 编辑冲突：注释明确说明 SwiftUI 的 onKeyPress 在子 TextField 获焦时不会冒泡到父层，因此"自定义回答"输入过程中按 Enter 不会误触发提交（与左右键翻页一致的焦点协作模型）
- 实现方案选择 `.onKeyPress(.return)` 而非 `keyboardShortcut(.defaultAction)`，集中管理键盘交互，避免 `.defaultAction` 在 TextField 编辑时的不可控行为

### 🟡 问题 2（死分支 onChange）— 已修复 ✓【处置 A】

代码位置：`AskQuestionCardView.swift:315-321`

```swift
/// index 越界的单点防御（getter 内 guard 兜底）。
/// questions 数组来自 `let request`，view 生命周期内不可变，因此不再
```

- `goPrev / goNext` 仍各自 `guard !isFirst / !isLast`，编辑 currentIndex 的两个入口都受保护
- 注释充分说明了"为何不再依赖 onChange"，符合 CLAUDE.md "注释解释 why" 约定
