## 2. 背景与目标

### 2.1 业务背景

cc-anywhere 是 Mac + Server + Android 三端协作的 Claude Code 客户端。当前手机端在以下两个场景体验不佳：

**场景 A**：Claude 通过 `AskUserQuestion` 工具一次问多个问题时（典型 3-5 个），手机端卡片纵向堆叠所有问题，6 寸竖屏上需要反复滚动。Mac 端已在前一个 L3 流程完成翻页改造，手机端尚未对齐。

**场景 B**：当 Claude 创建 Task 子 agent 干活时：
- 手机端**完全看不到子 agent 在干啥**（JSONLWatcher 显式过滤 `agent-*.jsonl` 文件）
- 子 agent 内部调 Bash/Write/Edit 触发的 `tool_approval` 卡片到手机端**没有任何上下文**（用户视角："这个 Bash 为啥要批准？")
- permission mode = default/plan 下子 agent 高频触发卡片，手机端容易感觉"卡住"

### 2.2 项目目标

| 编号 | 目标 |
|---|---|
| G1 | 手机端 AskUserQuestion 多问题卡片体验与 Mac 端对齐（单题翻页 + X/N 进度 + 左右滑动 + 上下题/提交按钮） |
| G2 | 手机端能完整看到 Task 子 agent 在干啥（默认折叠块 + 展开后含 thinking/tool_use/tool_result 序列） |
| G3 | 子 agent 内部触发的 tool_approval 卡片带 parent context（"⚡ 子 agent: <prompt 摘要>" chip 标签） |
| G4 | "卡住"现象修复（permission mode 继承父 tab + 手机端 pending 卡片队列 UI） |

### 2.3 成功指标

- 手机端在 N≥2 问题时一次只显示一题，可左右滑动 / 上下题翻页，回退保留答案
- Task 子 agent 创建后，手机端立即出现折叠块（标题为 Task prompt 摘要），点开看到完整 thinking 序列
- 子 agent 内部 tool_approval 卡片顶部有 chip 标识来源
- 当多个 pending 卡片同时存在，手机端显示"1/N 待审批"，答完自动切到下一个

## 3. 用户故事

### US-1：手机端单题翻页（对应 G1）

> 作为开发者，当我远离电脑用手机看 Claude 进度时，Claude 一次问我多个问题，我希望一次只看一题、答完一题再看下一题，可以来回切换修改之前的答案，最后一起提交。

### US-2：子 agent 进度可见（对应 G2）

> 作为开发者，当 Claude 创建子 agent 跑一个大任务（如批量重构）时，我希望在手机上看到子 agent 在干啥（"它在改第 3 个文件"），即便我不需要逐条读完所有 thinking，至少能展开看到运行轨迹，确认它没跑偏。

### US-3：tool_approval 上下文识别（对应 G3）

> 作为开发者，当子 agent 内部要跑 Bash 询问我是否批准时，我希望卡片告诉我"这是子 agent 在干活中的请求"+ 它的任务摘要，而不是孤立的"批准 Bash 吗"。

### US-4：高频卡片不卡死（对应 G4）

> 作为开发者，当子 agent 在高频跑工具时（一秒触发好几个 tool_approval 卡片），手机端不应该堆积卡死 — 而应该按队列依次展示，并显示"还有 N 个待审批"让我心里有数。

## 4. 功能清单

| 优先级 | 编号 | 功能 | 涉及端 |
|---|---|---|---|
| P0 | F1 | 手机端 AskUserQuestion 卡片单题翻页（X/N + 左右滑动 + 上下题/提交按钮 + 末题提交） | Android |
| P0 | F2 | mac 端 JSONLWatcher 解除 `agent-*.jsonl` 过滤，透传 sidechain 字段 | Mac |
| P0 | F3 | 手机端 ChatRepository 按 parentToolUseId 聚合子 agent 消息为折叠块 | Android |
| P0 | F4 | 手机端 SubAgentFoldedBlock widget（默认折叠 / 点开展开） | Android |
| P0 | F5 | tool_approval 卡片注入 subAgentSummary + parentToolUseId（来自父 session JSONL 反查） | Mac + Android |
| P1 | F6 | 手机端 pending 卡片队列 UI（顶部显示"1/N 待审批"，答完自动出队） | Android |
| P1 | F7 | mac 端 permission mode 继承验证（实测确认 / 必要时补代码） | Mac |
| P1 | F8 | 历史回放（HistoryBridge）也带 sidechain 字段，断线重连后子 agent 块正确显示 | Mac + Android |

### 范围明确：不做什么（YAGNI）

- ❌ 不支持子 agent 嵌套（孙 agent）— Claude Code 当前不支持，跟随其约束
- ❌ 不支持手机端从子 agent 折叠块跳转到父 Task tool_use 的"导航"功能 — 上下文 chip 文字已足够
- ❌ 不修改 mac 端 AskQuestionCard 翻页 UI（前一个 L3 流程已完成）
- ❌ 不动 Server（dumb proxy 哲学保留）
- ❌ 不引入端到端加密、不引入多租户

## 5. 非功能性需求

| 类型 | 要求 |
|---|---|
| 性能 | 单 tab 同时活跃子 agent 数 ≤ 3；折叠块展开渲染 < 200ms；翻页动画 < 300ms |
| 兼容性 | 协议字段均为 optional — 旧版手机 / 旧版 mac 不解析照常工作（退化为现有体验：子 agent 看不到） |
| 安全 | 自定义输入框仍保留 200 字符上限；子 agent 内容遵循现有 truncate 规则；不引入新的 PII 流出路径 |
| 架构哲学 | Server 零改动（dumb proxy 保留）；新协议字段通过透传机制实现 |
| 可用性 | 子 agent 折叠块在主对话流中视觉清晰（如左侧色条 / 缩进 / icon），不与普通消息混淆 |

## 7. 风险与依赖

| 编号 | 风险 | 严重度 | 缓解 |
|---|---|---|---|
| R1 | JSONL 字段名 / agent-*.jsonl 命名规则尚未实测确认 | 高 | 阶段四技术文档列实测前置任务作为开发任务 0；实测结果决定 parentToolUseId 提取路径 |
| R2 | 手机端聚合 race（子 agent 内容先于父 Task tool_use 到达） | 中 | 手机端 `pendingSidechainBuffer` 暂存机制 + 5 秒超时孤儿展示 |
| R3 | 子 agent 内容跟父 session.jsonl 的 Task tool_result 内容重复 | 中 | 实测确认 Claude SDK 语义 — 父 tool_result 只含 final output；如真重复 → 手机端按 message id dedup |
| R4 | 协议字段双端不一致导致解析错误 | 高 | mac 端 ProtocolMessage.swift 与 手机端 protocol_message.dart 在同一 commit 内同步改 |
| R5 | 队列 UI 与翻页 UI 视觉冲突 | 低 | 队列指示在卡片**顶部**（1/N 待审批），翻页指示在卡片**底部**（X/Y 题），维度不同 |

### 外部依赖

- Claude Code CLI 的 JSONL 落盘格式稳定（已运行 6+ 个月，无突变记录）
- Claude Code SDK 的 hook 协议兼容性（M1-M4 已验证）

### 降级方案

- 协议字段 optional 设计：旧版手机 / 旧版 mac 接收新字段静默忽略，退化为现有体验
- 单 commit 回滚：完整改动在 working tree → `git revert` 整段
