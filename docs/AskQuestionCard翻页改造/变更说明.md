# AskQuestionCard 多问题翻页改造 - 变更说明

> 文档版本：v1.0
> 级别：L3 - 小型变更
> 涉及范围：Mac 端单组件 UI 改造（手机端不在本次范围）

## 1. 变更内容

### 1.1 现状

`MacClient/Sources/CCAnywhere/Views/AskQuestionCardView.swift` 中的 `UserQuestionBody` struct 用 `ForEach` 将 Claude AskUserQuestion 工具返回的所有 `questions` **纵向堆叠**展示，底部一个统一的"提交"按钮：

```
┌─────────────────────────┐
│ Claude 想问您：         │
├─────────────────────────┤
│ 问题 1: ...             │
│   ◯ A  ◯ B  ◯ C  ◯ 其他 │
├─────────────────────────┤
│ 问题 2: ...             │
│   ◯ ... ◯ ... ◯ ...     │
├─────────────────────────┤
│ ...（N 个问题平铺）     │
├─────────────────────────┤
│              [ 提交 ]   │
└─────────────────────────┘
```

当 Claude 一次提出多个问题（每题再展开自定义输入框）时，卡片高度可达 1000+ px，超出舒适视野。

### 1.2 改造后

改为**单题翻页 UI**：一次只显示一个问题，通过底部步进按钮和键盘左右键切换：

```
┌─────────────────────────┐
│ Claude 想问您：    2 / 5│  ← 顶部进度指示
├─────────────────────────┤
│ 问题 2: ...             │
│   ◯ A                   │
│   ◯ B                   │
│   ◯ 自定义回答          │
├─────────────────────────┤
│  [← 上一题]  [下一题 →] │  ← 步进按钮（最后一题改为"提交"）
└─────────────────────────┘
```

### 1.3 具体改动点

| 改动 | 现状 | 改造后 |
|---|---|---|
| 问题渲染 | `ForEach` 平铺全部 | 仅渲染 `selections[currentIndex]` 对应的当前题 |
| 进度指示 | 无 | 顶部右侧文字 "X / N"（仅 N ≥ 2 显示） |
| 翻页按钮 | 无 | 底部「上一题」「下一题」按钮，首题禁用「上一题」，末题用「提交」替换「下一题」 |
| 键盘交互 | 无 | ← / → 翻页；Enter 在末题等价"提交"（仅 canSubmit 时） |
| 提交按钮 | 永远显示在底部 | 仅在最后一题显示，且需当前题已答完才启用 |
| 单题场景 | 无差别 | N=1 时自动隐藏所有步进按钮，行为等同现有体验 |
| 状态保留 | 现有 `selections: [Int: Set<String>]` 已按 index 存储 | 不变，翻页时自然保留所有题答案 |
| 提交逻辑 | 现有 `submit()` 一次性聚合所有答案 | 不变 |

## 2. 变更原因

### 2.1 用户反馈

实际使用中，Claude Code 通过 `AskUserQuestion` 工具一次性提出多个问题（典型 3-5 个，每个 3-4 个选项）的情况频繁。当前平铺布局导致：

- 卡片高度突破舒适视野（Mac 14 寸屏 1000px+ 已要求滚动）
- 用户需要在多个问题间反复来回扫视
- 一个长卡片视觉压迫感强，影响决策质量
- 自定义输入框展开后情况更糟

### 2.2 设计原则

- **单一焦点**：一次只关注一个问题，决策质量更高
- **可回溯**：允许用户答完后回到之前的题重新修改
- **零行为变化**：底层数据流（answers 聚合 + 一次性提交）完全不变，winner 锁仲裁逻辑、答案格式与现有完全兼容
- **退化兼容**：单题场景下视觉与现有完全一致，不引入额外控件

## 3. 影响范围

### 3.1 涉及文件

| 文件 | 改动类型 | 说明 |
|---|---|---|
| `MacClient/Sources/CCAnywhere/Views/AskQuestionCardView.swift` | 修改 | `UserQuestionBody` struct 状态机改造 |

### 3.2 不动的部分（明确边界）

- ❌ 不改协议（`ProtocolMessage.swift` / `AskQuestionPendingPayload`）
- ❌ 不改 `AskQuestionCardController`（提交入口 `submitUserQuestion(requestId:answers:)` 签名不变）
- ❌ 不改 `HookIpcServer`（winner 锁机制不变）
- ❌ 不改 hook bridge Python
- ❌ 不改手机端 `AndroidClient/lib/features/chat/widgets/ask_user_question_card_realtime.dart`（独立流程后续启动）
- ❌ 不改 `ToolApprovalBody`（工具批准分支不涉及多问题场景）

### 3.3 兼容性

- **数据兼容**：提交的 `answers: [String: String]` map 结构与现有完全一致，下游 hook bridge / Claude SDK 无感知
- **winner 锁兼容**：中途若手机端先答完触发 `AnsweredBanner` 显示，本卡片立即消失（现有逻辑）
- **多 Tab 兼容**：`currentIndex` 是组件内 `@State`，每个 Tab 的卡片各自一份，互不影响

## 4. 验证方式

### 4.1 功能验证（手动）

| 场景 | 验证步骤 | 预期 |
|---|---|---|
| 单题问题 | Claude 提 1 个问题 | 不显示 X/N 进度、不显示翻页按钮，提交按钮直接显示（与现有体验一致） |
| 多题正向翻页 | Claude 提 3 个问题 | 顶部显示"1/3"，底部"上一题"灰、"下一题"亮，点击下一题切到 2/3 |
| 多题回退 | 翻到 2/3 后点"上一题" | 切到 1/3，之前的答案保留 |
| 末题提交 | 翻到 3/3 | 底部按钮变为"上一题" + "提交"，三题全部答完才可点提交 |
| 键盘翻页 | 任意题按 ← / → | 翻页生效（首题 ← 无效、末题 → 无效） |
| 自定义输入项 | 任一题选"自定义回答"输入文本 | 切换题再切回，文本保留 |
| 中途被手机端答完 | 多题翻页中手机端先提交 | 本卡片立即消失，显示 AnsweredBanner |

### 4.2 回归验证

- `ToolApprovalBody`（工具批准卡片）UI 行为不变
- 工具批准卡片不显示翻页按钮（不涉及多问题）
- 中文输入法在自定义输入框内行为不变

### 4.3 视觉验证

- 暗色 / 亮色主题下进度文字、按钮颜色对比度正常
- 单题场景视觉与改造前完全一致（无多余控件）
- 卡片最大宽度仍为 520（不变）

## 5. 风险评估

| 风险 | 严重度 | 缓解 |
|---|---|---|
| 键盘事件冲突（与终端窗口/自定义输入框抢焦点） | 中 | 用 `.onKeyPress` modifier 限定卡片获焦时才生效；自定义输入框获焦时禁用翻页快捷键 |
| `@State currentIndex` 越界（题数为 0） | 低 | 现有 `canSubmit` 已校验 `questions` 非空；新增 currentIndex 边界 clamp |
| 翻页动画引入额外重绘成本 | 低 | 用 `.animation(.easeInOut(duration: 0.15), value: currentIndex)` 局部动画 |
| 用户答完 3/5 中途切到别的 Tab 再切回，currentIndex 复位 | 低 | `@State` 由 SwiftUI 自动持久化在 View 生命周期内（与现有 selections 一致），无额外处理 |
