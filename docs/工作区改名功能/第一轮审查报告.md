## 检查清单结果

- [x] **功能与变更说明 §1 一致（trim / 空回退默认 / 长度截断 40）**：`TabManager.swift:82-98` 完整实现：`trimmingCharacters(in: .whitespacesAndNewlines)` → 空 ⇒ `folder.lastPathComponent` → >40 ⇒ `String(trimmed.prefix(40))` → 否则原样。与文档逐条对应。
- [x] **边界值实现正确性**：
  - 空字符串：`trimmed.isEmpty` 命中 → 回退默认 ✅
  - 纯空白（含 `\t\n\r`）：`whitespacesAndNewlines` 包含全部 Unicode 空白与换行 ✅
  - 超长截断：`trimmed.count`（Character 计数）+ `prefix(40)`，对中文/emoji 按用户感知的"字符"截断而不是按 UTF-16 code unit，符合"40 字符"语义 ✅
  - emoji / 中文：String.prefix 按 Character 切片，emoji ZWJ 序列也会被当作 1 个 Character，不会被切坏 ✅
- [x] **`tabs[idx].name != resolved` guard**：第 93 行避免重复 persist 与广播。当用户点确定但内容未变（包括"trim 后等同原值"和"超长截断后等同原值"）时，不会触发无谓的磁盘写入和 ws 广播 ✅
- [x] **`changes.send(.renamed(tabs[idx]))` 外部可订阅**：`TabSyncBridge.swift:37-42` 订阅 `tabManager.changes`，第 73 行 `case .renamed(let t): summary = tabManager.summary(of: t); action = "renamed"`，第 77 行发送 `ProtocolMessage(type: "tab.changed", ...)`。外部链路完整可达。
- [x] **NSAlert Modal 阻塞主线程风险**：`startRename` 在主线程同步 `runModal()`，符合 AppKit 约定。CLAUDE.md §"Hook 软失败原则"仅禁止 hook bridge 让 UI 卡死或长时间阻塞；用户主动触发的短时模态对话框（与 `createTab`/`confirmAndClose` 一致）属于允许范畴。Modal 不影响 hook bridge IPC（Unix socket 在独立队列）和 ws（独立 actor），无主线程依赖问题 ✅
- [x] **同步到 Android 链路**：`renameTab` → `changes.send(.renamed)` → `TabSyncBridge.sink` → `sendTabChanged` → `ws.send("tab.changed")` → Server dumb proxy → Android。链路存在且与文档 §4 "同步到 Android" ≤2s 的期望相符。
- [x] **`TabError.nameTooLong` 残留调用**：`grep -rn "nameTooLong"` 全仓三处命中：定义（Tab.swift:83）、描述（Tab.swift:93）、**仍被 `createTab` 在第 53 行 throw**（name.count > 50 时）。即枚举并未死代码，`createTab` 仍依赖它。变更说明 §3.1 表格中"`TabError.nameTooLong` 不再抛出（保留枚举不影响调用）"的表述与实际不完全一致——`renameTab` 路径已确实不再抛，但 `createTab` 路径仍在抛。不过这并不是 bug，因为枚举本来就保留了，调用方 `createTab` 也没有改，行为兼容。**信息项，不阻塞**。
- [x] **TabStripView 原有"重命名…"行为一致**：`TabStripView.swift:91-103` 的 `startRename` 同样调用 `tabManager.renameTab(tab.id, to: input.stringValue)`，因此自动继承 trim / 空回退 / 截断 40 的新规则。与文档 §3.1 表格"已有的 startRename 不变（renameTab 行为升级后自动受益）"一致 ✅。

## 发现的问题

### 🔴 阻塞（必须修复才能进入第二轮）

无。

### 🟡 建议（可选修复）

1. **`TabError.nameTooLong` 文案不一致** — `Tab.swift:94` 的描述仍是 "Tab 名称最长 50 字符"，但 `createTab` 仍按 50 抛出（renameTab 则改成 40 不抛错）。这并非本次变更引入的问题，但当前注释/文档把"长度上限改为 40"作为新规则，存在两套阈值：`renameTab=40 截断`、`createTab=50 抛错`。可以选择在后续 review 中决定是否统一为 40 + 截断（与 renameTab 行为对齐），从而彻底移除 `nameTooLong` 抛出点；本轮不阻塞。
   - 文件：`MacClient/Sources/CCAnywhere/Services/TabManager.swift:49-56`、`MacClient/Sources/CCAnywhere/Models/Tab.swift:83,93-94`

2. **`SidebarView.startRename` 与 `TabStripView.startRename` 文案分歧** — Sidebar 的弹框 messageText 是"重命名工作区"且带 `informativeText` 提示"留空可恢复为默认名"；TabStrip 的是"重命名 Tab" 且无提示。功能等价，但文案差异会让用户感觉两条入口"不是同一个动作"。建议第二轮考虑统一文案/抽出辅助函数。本轮不阻塞。
   - 文件：`SidebarView.swift:143-156` vs `TabStripView.swift:91-103`

### ⚪ 信息（无需动作）

1. `renameTab` 的 `throws` 签名保留但**实际从不 throw**（persist 用 try，但当前实现里 persist 自身可能抛 `TabError.fileSystem`，被 `renameTab` 透传出去——所以 `throws` 并非"完全无意义"，磁盘失败仍会抛）。调用方两处都用 `try?` 吞掉了，这是 SwiftUI handler 中可接受的处理方式。注释里"实际上从不 throw" 略有误导（磁盘写入失败仍可能抛），但不影响正确性。
   - 文件：`TabManager.swift:81, 95`
2. `changes` 是 `PassthroughSubject` 而非 `CurrentValueSubject`，意味着新订阅者不会拿到"最近一次 rename"事件。这是预期行为（结构变更走 `sendFullList`），不影响 rename 同步。
3. `tabs[idx].name != resolved` 的 guard 在"用户改完但实际未变化"时直接 return（无日志），调试时可能不知道用户其实点了确定。可在第二轮考虑加 debug 日志。
