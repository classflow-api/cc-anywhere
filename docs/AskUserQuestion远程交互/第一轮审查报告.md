## 总结

### 4 个 🔴 阻塞问题（必须修复才能上线）

1. **#1 multiSelect 字段名 snake_case vs camelCase 不一致**：Mac 序列化 `multi_select`，Phone 读 `multiSelect`，导致多选题在 phone 端完全失效。这是协议层最直接的破坏，必须三端对齐。
2. **#2 R-F2-004 / R-F4-004 截断未实现**：Mac App `HookIpcServer.handleProgressPre` 与 tool_approval 路径都没有在 Mac 端做 200/500 字符截断，违反带宽与隐私保护硬规则。需要在 ws 推送前对 `tool_input` 长字段统一截断。
3. **#3 enableRemoteHook 默认值与 PRD 矛盾**：实现默认 OFF，PRD 默认 ON。导致 G4 验收与 F6-S1 首次许可弹窗流程触发不上，"开箱即用"体验缺失。需要二选一调整。
4. **#4 `unknown tab_id` / `bad request` 软失败语义错误**：当前实现返回 `error` 给 hook bridge，导致 Claude SDK deny；正确做法应输出 `{}` 让 SDK 走 fallback，符合 NFR-U1/U2"永远不能比没装 hook 更糟"硬规则。

### 6 个 🟡 警告

集中在：notification 字段映射可疑（#5）、超时后未清 hookPushedToolUseIds（#6）、winner 锁注释完善（#7）、command 路径双引号转义（#8）、缺 ask.tool_approval.answer 路由 + phone UI（#9，影响 M4 验收）、parseToolResponse 缺省字段语义（#10）。

### 2 个 🟢 建议

socket chmod 时序优化（#11）、Phone chat_repository hook dedup 路径简化（#12）。

### 整体结论

代码主流程完整、actor 串行化与 socket 框架选型可靠、winner 锁与超时回收逻辑正确。但 4 个阻塞问题涉及**协议字段名层面破坏多选题功能**、**关键的安全/隐私截断规则未实现**、**默认行为与 PRD 矛盾**、**软失败语义错误**，必须在进入第二轮 Review 前由开发者修复。修复完成后再走第二轮（边界值与业务影响）。
