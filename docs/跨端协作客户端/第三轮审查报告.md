## 各维度详细发现

### 1. 代码质量

#### 命名与结构
- 三端命名规范统一：Mac 服务 `XxxManager` / `XxxBridge` / `XxxHost`、Server `XxxService` / `XxxRouter`、Android `XxxRepository` / `XxxClient`。一致清晰。

- 三端协议消息常量（`TypeBind` / `ProtocolType.bind` / `"bind"`）取值一致，三端 `*messages.go` / `ProtocolMessage.swift` / `protocol_message.dart` 通过文件 header 注释互引。

#### 字符串硬编码
- 所有错误码（`INVALID_TOKEN` 等）已抽常量，三端齐全。
- WebSocket 路径 `/ws`、HTTP 路径 `/upload/` `/download/` `/healthz` 散落在 server 路由 + Mac/Android client 代码中，无单一常量来源。**🟢 建议**：抽到 `protocol/paths` 共享常量（跨语言难，但至少在 Server 内集中）。
- MacClient `DeviceManager.swift:120-125` QR payload 字段名 `"v"`、`"server"`、`"port"`、`"sub_token"` 直接硬编码字符串字典，Android `server_config.dart:64-65` 解析端也硬编码同样的字符串。两侧需保持同步，但无单元测试覆盖，未来字段调整易出 bug。**🟢 建议**：定义 `QRPayload` Codable struct（Mac）+ class（Dart），共享 JSON shape。

#### 重复代码
- Server `protocol.MustEnvelope` / `protocol.NewEnvelope` 大量出现在 server.go 中，每次都用 `_` 忽略 marshal error；可以 wrapper（已部分实现但 server.go 内调用点未用）。**🟢 建议**：在 connection.go 加 `conn.SendType(type, data)` 便利函数。
- MacClient 三处独立 helper（DeviceManager / TabSyncBridge / HistoryBridge 各自）做 `Encodable → AnyJSON` 转换：
  - `DeviceManager.swift:45-46` `JSONEncoder().encode(payload) → JSONDecoder().decode(AnyJSON.self)`
  - `TabSyncBridge.swift:82-83` 同样链
  - `HistoryBridge.swift:57-60` 同样链
  - `InputInjector.swift:90-96` 又重复了一遍（用 `JSONSerialization`）
  **🟢 建议**：抽 `AnyJSON.encode<T: Encodable>(_ p: T) -> AnyJSON?` 静态方法去重。

#### 可读性
- WSClient.swift `handleIncoming` 内 switch 大量字面字符串 `"bind.ack"` / `"force_disconnect"` 等。与 ProtocolMessage 模型未对齐（Server / Android 都有 TypeXxx 常量）。**🟢 建议**：在 Mac 端新建 `ProtocolType` 枚举或 static let 常量集。
- JSONLWatcher `WatchStream` 嵌套于同文件且不公开使用，但作为内部类型已可读。

### 2. 错误处理

#### 真处理 vs 吞错误
- Server 端 `_, err := s.db.ExecContext(...)` 在 `device.go:96` BindPhone 过期分支用 `_, _ = s.db.ExecContext(...)`（line 96），**双 `_` 静默吞错**。生产中 DB 写失败（磁盘满 / 锁超时）需要至少 WARN 日志，否则过期-revoke 转换可能永远失败而无人知。
- MacClient `JSONLWatcher.swift:209-217` 多次 `try?` 链：`FileHandle(forReadingFrom:)` → `seek(toOffset:)` → `readToEnd()`。读失败时整行/整批被丢且无日志。**🟡 警告**：见【发现的问题 - 🟡 #4】。
- MacClient `Logger.swift:81` `fileHandle = try? FileHandle(forWritingTo: ...)`；日志文件打开失败时**所有 log 静默丢失**，仅 os_log 仍走 Console.app。生产中如果 Logs 目录权限有问题（macOS Sandbox 异常），用户从 LogViewerWindow 看到的会是空。**🟡 警告**：见【发现的问题 - 🟡 #5】。
- AndroidClient `chat_repository.dart:158` 在 sendTextWithImages 内拿文件大小 `try { sizeBytes = await file.length(); } catch (_) { /* 忽略 */ }` —— 静默吞错，长度获取失败时 attachmentSizeBytes 为 null，UI 显示"未知大小"。可接受。

#### 错误消息质量
- Server 错误消息基本含上下文：`"create upload file"`、`"mac is offline"`、`"sha256 mismatch"`。
- Mac `WSClient.swift:97` `"send dropped (no task): \(message.type)"` 清晰。
- Mac `JSONLWatcher.swift:154` `"FSEventStreamCreate failed for \(directory.path)"` 含路径。
- Android `ImageUploadException` / `BindFailedException` 都带 code + message，下游 catch 后 SnackBar 可见。

#### panic / fatalError / 强解包
- Server `protocol/messages.go:128` `MustEnvelope` 显式 panic；仅用于已知合法 payload，可接受。
- Server `main.go:30` `os.Exit(1)` 由 cobra 调用，合理。
- MacClient 全代码无 `fatalError`、无 `try!`、无 `as!`；仅 `WSClient.swift:69` `session!.webSocketTask(...)` 一处强解包，但 session 在前一行刚被赋值，无 nil 风险。
- MacClient `Logger.swift:67` 与 `PreferencesService.swift:21` 各有一处 `urls(for:).first!`；macOS 上 ApplicationSupport / Library 必存在，可接受。
- AndroidClient 无 force-unwrap，但 `chat_repository.dart` 中多处 `m.data['tab_id'] as String?` 在 null 时直接 return，符合 fail-safe 原则。

### 4. 安全性

#### 自签证书信任
- MacClient `ServerConfig.swift:15` `trustSelfSigned: Bool = false`（默认拒绝）。✅
- WSClient `urlSession(_:didReceive:)` 仅在 `trustSelfSigned == true` 时 `useCredential(URLCredential(trust: trust))`。无验证就接受任何证书是有意为之，符合 R-M5-03。

- **MacClient `WSClient.swift:225-235` URLSessionDelegate 通过 `MainActor.assumeIsolated` 读取 `currentConfig?.trustSelfSigned`**，nonisolated → main actor 跨界访问，逻辑上读取的是即时值，可接受。

### 5. 性能

#### 大循环 / N+1
- Server `device.go:196-217` List 单次 SELECT 拉全部 sub_tokens（无分页）；个人工具单 user 设备数 ≤ 10，无 N+1 风险。
- Server `hub.go:114-118` BroadcastToPhones for-loop 同步 send；每个 send 含 1s send timeout（`connection.go:72-76`），慢消费者会被踢。`PhoneConns()` 返回 snapshot slice 避免读锁期间业务调用。✅

#### 大消息序列化
- Server `protocol/messages.go:135` DecodeData 用 json.Unmarshal 解码 RawMessage；按需。
- MacClient `MsgStreamBridge.swift` (DependencyContainer.swift:108-126) 把整批 ParsedMessage[] → AnyJSON[] → JSONEncoder.encode → 单次 `t.send(.string(...))`。**当一批 N 条消息每条几 KB 时整 JSON 字符串可达数 MiB**，相对于 4 MiB Server 限制有压力。第二轮 🟡 #2 已识别 readLimit；这里同时是发送方一次性 marshal 的内存峰值。**🟡 维持**。

#### PTY 字节流转发
- ProcessHost.write → `view.process.send(data: ArraySlice(bytes))` — SwiftTerm 内部异步写 PTY；调用方不阻塞。✅
- 输出方向：PTY → SwiftTerm 内部 buffer + 渲染，cc-anywhere 不参与。

#### JSONL 全量读取
